# 三合一 ASR VAD 断句边界与尾音隔离
Status: ready-for-agent

## Problem Statement

设备通过统一 WebSocket 发起语音模式的 Agent Application 会话时，三合一链路负责把设备 PCM 音频交给 ASR，并在得到最终转写后继续执行 LLM 和 TTS。

当前上游 ASR 已通过 VAD 发送“检测到停说”的事件后，服务端仍持续接受并转发设备 PCM，直到稍后收到最终转写事件才停止。设备因此继续显示录音状态；用户在这段等待时间内无意说出的 1–2 个字可能被录入，造成 Device Chat Contract 中的最终问题文本不符合原始意图。运行中的静音断句时长为 1000ms，也会使对话至少等待一秒才可能进入最终转写阶段。

该问题发生在语音模式的 Agent Application 三合一流程中，不应通过新增专用 WebSocket URL、绕过设备码认证或修改 Agent Runtime Backend 的选择逻辑来解决。

## Solution

当上游 VAD 明确报告当前发言结束时，服务端立即关闭该 ASR 会话的音频接收闸门，并通过统一 WebSocket 向设备发送一个独立的、供应商无关的“输入已停止”事件。设备收到该事件后立刻停止麦克风采集、PCM 上送和录音 UI，但保持当前 WebSocket 连接，等待该发言的最终转写。

服务端仍需等待 ASR 的最终转写，之后才结束 ASR 阶段并启动 LLM/TTS：这避免在 VAD 边界提前关闭上游会话而丢失最终文本。任何在音频闸门关闭后到达的 PCM 必须被服务端丢弃，即使旧版设备尚未实现新的停止采集事件也不能夹带尾音。有效最终转写继续按既有 Device Chat Contract 驱动一次 LLM 与 TTS；空结果或已过滤的语气词结束会话但不触发 LLM/TTS。

全局 ASR 对话断句配置采用 400ms 作为推荐部署值，以缩短正常对话的端点等待；VAD 阈值继续保留为可按现场噪声调节的配置，不把单一阈值值硬编码为通用答案。

## User Stories

1. 作为设备使用者，我希望说完一句话后录音状态立即结束，以便知道系统已开始处理我的问题。
2. 作为设备使用者，我希望说完后的随口补充不会混入上一句，以便智能体按我的原始问题回答。
3. 作为设备使用者，我希望最终 ASR 文本只对应一次完整发言，以便不会收到由尾音造成的错误回答。
4. 作为设备使用者，我希望在停止说话后仍能等待识别结果，以便不必手工重复发送或重新唤醒设备。
5. 作为设备使用者，我希望 ASR、LLM 与 TTS 仍按原有顺序工作，以便语音问答体验不因修复断句而改变。
6. 作为设备运行时客户端，我希望收到明确的输入停止事件，以便能同步停止麦克风采集、PCM 上送和录音动画。
7. 作为设备运行时客户端，我希望在输入停止后继续接收最终转写、LLM 和 TTS 事件，以便不会将停止采集误解为会话取消。
8. 作为运行旧版本客户端的设备，我希望服务端仍拒绝断句后的 PCM，以便服务端单独保证尾音不会进入 ASR。
9. 作为 Agent Application 使用者，我希望有效最终转写只触发一次 LLM 请求，以便不会形成重复回答或重复 TTS。
10. 作为 Agent Application 使用者，我希望被过滤的语气词不会触发 LLM/TTS，以便不会为无意义输入生成回答。
11. 作为设备使用者，我希望空识别结果能回到可继续交互的状态，以便不会卡在“正在录音”或“正在识别”。
12. 作为平台管理员，我希望对话场景使用较短的推荐静音断句时长，以便减少正常停说后的等待。
13. 作为平台管理员，我希望仍可按现场噪声调节 VAD 阈值，以便平衡漏检与背景噪声误判。
14. 作为平台管理员，我希望 VAD 参数保留现有范围校验，以便不会配置出供应商不支持的值。
15. 作为设备运维人员，我希望新事件保留请求与链路追踪标识，以便能关联一次设备语音会话的 ASR、LLM 和 TTS 阶段。
16. 作为公司用户，我希望本公司设备的语音会话仍只使用自身的 Agent Application 和数据，以便修复不影响租户隔离。
17. 作为设备运行时客户端，我希望继续只连接统一 WebSocket 入口，以便不必维护第二条 ASR 专用连接。
18. 作为平台维护者，我希望上游供应商事件不会直接泄漏为设备协议，以便 Device Chat Contract 维持稳定、可替换的实现边界。
19. 作为平台维护者，我希望取消会话仍可立即中止 ASR、LLM 和 TTS，以便用户主动取消的优先级高于等待最终转写。
20. 作为平台维护者，我希望设备在 VAD 停说和最终转写之间发送的多帧 PCM 都被拒绝，以便修复不依赖某一个特定的尾音长度。
21. 作为平台维护者，我希望 ASR 替换词继续仅作用于最终文本，以便断句修复不改变既有租户级文本修正规则。
22. 作为使用第三方聊天机器人的公司，我希望语音识别边界修复不改变第三方会话的会话 ID 与上下文复用，以便 Agent Runtime Backend 的既有行为保持不变。

## Implementation Decisions

- 本需求只覆盖统一 WebSocket 中语音模式的 Agent Application 三合一流程：设备 PCM → ASR → LLM → TTS。文本模式的 Agent Application 会话和独立 ASR 测试会话保持原有语义。
- 服务端把上游 ASR 的 VAD 停说事件视为“关闭设备音频输入”的边界，而不是最终转写完成的替代信号。每个 ASR 会话只允许第一次有效停说事件关闭闸门，之后的二进制 PCM 一律忽略。
- 服务端新增设备侧事件 `asr.input_stopped`。事件使用当前会话的命令标识、请求标识和链路追踪标识，并以 `reason: "vad"` 表示由 VAD 确认输入结束。该事件是 Device Chat Contract 的供应商无关扩展，不透传上游事件名或原始负载。
- 设备收到 `asr.input_stopped` 后必须立即停止麦克风轨道或采集器、停止 PCM 上送并结束录音 UI；不得把该事件当作取消或断开 WebSocket。设备继续等待 `asr.transcript`、`asr.done`、LLM 和 TTS 事件。
- 服务端关闭音频闸门后仍保持上游 ASR 会话，等待该已提交发言的最终转写。不得仅因 VAD 停说就向上游发送会导致最终文本截断的强制会话结束命令。
- 收到有效最终转写后，保持现有顺序：下发最终 `asr.transcript`、收束 ASR、下发 `asr.done`，随后只启动一次 LLM；LLM/TTS 的流式与 Device Runtime Conversation 记录规则不变。
- 若最终事件为空或在启用语气词过滤后没有有效文本，服务端必须收束 ASR 并下发 `asr.done`，但不得启动 LLM/TTS，也不得把先前的预览文本当作最终问题发送。
- 主动取消在所有阶段都优先：取消命令要关闭音频闸门、停止上游 ASR，并取消后续尚未启动或正在运行的 LLM/TTS，不再发送输入停止后的正常完成事件。
- 断句后的 PCM 丢弃必须由服务端保证；Android 客户端停止采集是降低无效流量和改善 UI 的协同措施，不能成为数据正确性的唯一防线。
- 全局 ASR 的推荐 `vadSilenceDurationMs` 设为 400ms，并通过既有 ASR 配置能力部署；保留现有 200–6000ms 的合法范围。现有 `vadThreshold` 保持可配置，在真实设备环境中基于噪声测试调整，不在本需求中强制改成固定值。
- 继续使用现有设备码解析、`X-Device-Code` 相关运行时授权、租户解析和统一 WebSocket 入口；不新增 REST 路径或业务 WebSocket URL。
- Agent Runtime Backend 的平台 LLM 与 Third-Party Chatbot Interface 继续共享稳定的 Device Chat Contract。第三方上游会话 ID 的保存与复用不因 ASR 边界事件改变，符合 ADR-0003。

## Testing Decisions

- 采用现有统一 WebSocket 的三合一 ASGI 集成测试作为最高且主要的测试缝隙。测试从设备发起语音模式 Agent Application 会话，到观察设备事件、上游 ASR 收到的 PCM、以及后续 LLM/TTS 事件；不直接断言内部布尔字段或私有辅助函数。
- 新增一条最小回归场景：设备先上送正常 PCM；模拟上游 VAD 发出停说；断言设备收到一次 `asr.input_stopped`；随后上送多帧尾音 PCM；断言上游收到的音频帧数不再增加。
- 同一条最高缝隙测试继续模拟最终有效转写和上游 ASR 完成，断言最终转写、`asr.done`、一次 LLM 请求、一次流式 TTS 链路及最终 `agent.done` 的顺序保持正确。
- 增加空结果与已过滤语气词的回归场景：VAD 停说后仍关闭音频闸门并完成 ASR，但不触发 LLM/TTS，也不遗留可继续转发 PCM 的会话。
- 覆盖取消竞争：输入停止事件后、最终转写前发送取消，断言不启动 LLM/TTS，且不再接受 PCM。
- 扩展 ASR 配置 API 测试，确认 400ms 推荐部署值符合既有合法范围；保留阈值和静音时长的边界校验。
- Android 运行时需要做协议验收：收到 `asr.input_stopped` 后停止采集与录音 UI，但不关闭统一 WebSocket，并继续正确处理最终结果和后续 TTS。
- 验证应优先在 Docker Compose 的 backend 容器执行目标 Django 测试并使用 `--keepdb`；本次若触及 Web 管理页或协议类型定义，前端至少执行生产构建。

## Out of Scope

- 不更换 ASR 供应商、模型或基础 WebSocket 地址。
- 不新增客户端本地 VAD、唤醒词逻辑、音频降噪或回声消除算法。
- 不改变 ASR 替换词、语气词词表、租户隔离、设备码授权或 Agent Application 权限规则。
- 不新增 REST API、第二条 WebSocket URL，或把上游供应商事件直接暴露给设备。
- 不重构 LLM、TTS、第三方聊天机器人上下文持久化或 Device Runtime Conversation 的数据模型。
- 不把 VAD 阈值固定为某一个适用于所有现场的值；阈值调优以真实设备环境测试为准。

## Further Notes

- 已验证的最小复现为：上游 VAD 已报告停说后，服务端仍接受并转发尾音 PCM。该信号连续三次稳定复现，当前已有测试只覆盖“最终转写后停止”，未覆盖 VAD 停说与最终转写之间的边界。
- 供应商 VAD 的静音断句表示语句结束；对话场景采用 400ms 可缩短等待，但不取代服务端关闭音频闸门这一必需的正确性保障。
- 本需求不与现有 ADR 冲突：它只扩展稳定的设备侧实时事件，并保持统一 WebSocket 和独立 Agent Runtime Backend 的边界。
