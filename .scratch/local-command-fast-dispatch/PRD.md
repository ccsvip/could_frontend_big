# 公司本地指令快速分发

Status: ready-for-agent

## Problem Statement

公司用户已经在指令管理中维护了可执行的控制指令，但设备通过统一 WebSocket 发起 Agent Application 会话时，指令识别仍先将该公司全部启用指令作为 OpenAI Tool 发送给上游 LLM。当前“杨老板的公司”有 110 个启用控制指令；对于“帮我打开杨荣亮的电脑”这类明确指令，上游 Tool Calling 首轮出现 20 秒超时和 HTTP 503，服务端随后静默回退到普通 LLM 对话，最终返回与设备控制无关的隐私拒答。

这使 Company User 无法可靠地控制已配置设备，也使明确的本地指令比普通对话更慢。当前 Device Chat Contract 保持连接和 TTS 流正常，但没有告诉客户端指令识别已失败或已退化为普通对话。

## Solution

为公司本地指令引入优先于 LLM 的快速分发路径。系统先在当前公司的启用控制指令与任务指令中进行本地候选匹配；当只有一个高置信候选时，直接异步执行该指令，并通过既有统一 WebSocket 返回执行结果和可播报的简短反馈，不再发起 Tool Calling 首轮或第二轮自然语言回复。

当本地匹配不能安全地确定唯一指令时，系统仅将有限候选作为 Tool 提供给平台 LLM；普通问题直接走现有 Agent Application 对话，不先等待全量 Tool Calling。候选选择或 Tool Calling 暂时不可用时，不得把明确的公司指令悄然降级为普通聊天拒答。系统应返回可定位的指令分发状态，同时保留普通对话的现有体验。

## User Stories

1. 作为 Company User，我希望说出已配置的明确控制指令后立即触发对应设备动作，以便不必等待普通 LLM 回答。
2. 作为设备使用者，我希望“帮我打开杨荣亮的电脑”能命中本公司的对应电脑控制指令，以便完成现场设备控制。
3. 作为设备使用者，我希望成功下发后收到简短、可直接播报的执行反馈，以便确认设备已接收动作。
4. 作为设备使用者，我希望设备不可达或下发失败时收到明确失败原因，以便判断是否需要重试或检查设备。
5. 作为 Company User，我希望普通问题继续使用 Agent Application 对话，以便本地指令优化不会改变问答能力。
6. 作为 Company User，我希望不明确的说法不会任意触发相似设备，以便避免错误控制。
7. 作为 Company User，我希望同一公司只有一个高置信候选时可以容忍轻微的输入差异，以便语音识别和常见错别字不会阻断控制。
8. 作为 Company User，我希望多个候选同样合理时系统保持安全，不自动下发任一设备指令。
9. 作为 Company User，我希望本公司的指令只在本公司的 Device Runtime Conversation 中可见和可执行，以便保持公司数据隔离。
10. 作为设备运行时客户端，我希望继续使用统一 WebSocket 的 `agent.session.start`、`llm.delta`、`llm.done` 和 `agent.done` 事件，以便无需实现新的业务连接。
11. 作为设备运行时客户端，我希望在本地指令命中时从既有完成事件中得到结构化分发结果，以便展示或记录执行状态。
12. 作为 Company User，我希望明确指令不再因上游 Tool Calling 超时而收到无关的安全拒答，以便设备控制的行为可预测。
13. 作为平台维护者，我希望普通对话不再携带公司全部指令到上游，以便降低请求体大小、等待时间和上游故障影响面。
14. 作为平台维护者，我希望有限候选的 LLM 选择仍可处理不能本地确定的自然语言表达，以便保留必要的语义识别能力。
15. 作为平台维护者，我希望 Tool Calling 超时、503 和无候选等结果被区分记录，以便能定位本地匹配、上游选择和设备下发分别出现的问题。
16. 作为平台维护者，我希望控制指令的 UDP/TCP 下发继续为非阻塞 I/O，以便一个设备动作不会阻塞其他实时会话。
17. 作为平台维护者，我希望命中本地指令后不再为生成修辞性回复额外请求一次 LLM，以便缩短首个有效反馈时间。
18. 作为平台维护者，我希望任务指令沿用其已有的子步骤执行语义，以便快速路径不会改变任务编排结果。
19. 作为平台维护者，我希望现有 Device Runtime Conversation、会话 ID 和 Agent Runtime Backend 边界保持不变，以便普通会话上下文不受影响。
20. 作为平台维护者，我希望前端调试控制台能看见命中、未命中、歧义和上游降级等分发结果，以便现场验收无需查看服务器日志。

## Implementation Decisions

- 本需求覆盖平台 LLM 类型的 Agent Application 设备会话及现有独立 LLM 会话；Third-Party Chatbot Interface 继续按其既有的 Agent Runtime Backend 行为处理，不向第三方接口透传本地工具。
- 指令选择必须先按当前 Company 的租户作用域查询启用的控制指令和任务指令；不得通过请求参数扩大 Company User 的可见范围，也不得将其他公司的指令带入候选。
- 快速路径以现有指令名称、指令码及其标准化文本构建候选。标准化仅处理大小写、空白和标点等无语义差异；对轻微语音识别/错别字差异可做受限相似度候选，但只有唯一高置信候选才允许自动执行。
- 候选为零时立即进入现有普通 LLM 对话，不进行全量 Tool Calling 预检。
- 候选唯一且达到自动执行阈值时，直接调用既有异步 UDP/TCP 指令执行器或任务步骤执行器。执行成功、设备下发失败和超时都生成固定且可播报的中文结果；不再为该结果进行第二轮 LLM 生成。
- 候选存在但不唯一或未达到自动执行阈值时，最多将有限候选作为 Tool 发送给平台 LLM。不得将公司全部启用指令发送给上游。
- 有限候选的 Tool Calling 失败、超时或上游 5xx 时，必须以可观测的分发结果结束该候选分支；明确候选不得静默回退为普通 LLM 的内容回答。没有候选的普通问答仍可按现有策略调用 LLM。
- 继续使用现有异步模型：Agent 会话通过异步任务运行，数据库读取经现有异步边界执行，UDP/TCP 下发保持非阻塞。优化目标是减少串行上游 LLM 往返，而不是引入新的同步调用。
- 继续使用现有统一 WebSocket 入口。`llm.done` 的现有 `commandDispatch` 结构成为命中、未命中、歧义、上游选择失败和执行失败的稳定观测点；`agent.session.start` 经既有复用路径得到同样结果。
- Device Chat Contract 的正常 `llm.delta`、TTS 片段、`llm.done` 和 `agent.done` 顺序保持稳定。快速路径的固定执行反馈必须照常进入 Agent Application 的 TTS 队列和 Device Runtime Conversation 记录。
- 指令管理的数据模型、控制指令分组、已有指令数据迁移和设备网络地址不在本需求中改动。
- 需要清理当前会将 Tool Calling 异常吞掉后无差别回退普通聊天的行为；日志应记录 Company、候选数、分发路径、上游错误类别和设备执行结果，但不得记录 API Key 或其他凭据。

## Testing Decisions

- 主要测试缝为现有统一 WebSocket 的 `agent.session.start` ASGI 集成测试。这是用户实际触发的最高层行为缝，应断言外部事件和设备下发结果，而不是私有辅助函数或任务对象。
- 新增一个公司、设备、Agent Application 和唯一控制指令的最小场景。发送明确问题后，断言收到正常会话完成事件、结构化命令分发结果和一次异步 UDP/TCP 下发；同时断言无需调用 Tool Calling 或第二轮回复生成。
- 新增轻微文本差异但唯一高置信候选的场景，验证系统能命中预期指令；新增两个相近候选的场景，验证系统不会自动下发任一指令。
- 新增无候选普通问题场景，断言仍只走一次现有普通 LLM 对话，且不会因指令预检增加可感知的 Tool Calling 等待。
- 新增有限候选 Tool Calling 返回工具调用的场景，验证只传入候选集、正确执行并返回完成事件；新增 Tool Calling 超时与 503 场景，验证明确候选不会变成通用安全拒答，且错误状态可由客户端观察。
- 新增控制指令下发超时和网络错误场景，验证 Device Chat Contract 给出失败结果而不伪报成功。
- 新增任务指令场景，验证快速路径沿用现有子步骤执行、失败汇总和租户隔离。
- 扩展既有指令工具和实时命令分发测试作为低层回归保护；已有统一 WebSocket 设备会话测试作为上层回归保护。测试在 Docker Compose 的 backend 容器内使用 `--keepdb` 执行。
- 前端调试控制台的结构化分发状态采用现有 WebSocket 负载验证；若修改 TypeScript 类型或显示逻辑，执行前端生产构建。

## Out of Scope

- 不更换或配置上游 LLM 供应商、模型、API Key、超时策略以外的基础设施。
- 不承诺上游 LLM 对任意自然语言或任意错别字的理解；自动执行只适用于唯一高置信候选。
- 不新增第二个 WebSocket URL、动词式 HTTP API 或设备侧认证方式。
- 不重做指令管理页面、批量导入、分组设计或已有指令数据的业务分类。
- 不修改 UDP/TCP 目标地址、端口、协议和值编码，也不放宽设备网络访问的安全边界。
- 不改变 Third-Party Chatbot Interface 的会话 ID、上下文复用或回答路径。
- 不把本地指令快速路径扩展为通用工作流、权限提升或跨公司设备控制能力。

## Further Notes

- 已确认的生产式开发环境证据：`agent.session.start` 会异步进入统一 LLM 会话路径；租户 12 的“帮我打开杨荣亮的电脑”在 Tool Calling 首轮记录为 `LLM 请求超时`，随后回退为普通 `qwen-flash` 拒答。相同租户的明确控制指令也记录过 Tool Calling HTTP 503。
- 当前 Company 有大量启用指令，说明“把全量工具发送给上游再决定是否控制”既不是可靠的控制策略，也不符合明确指令应快速响应的产品预期。
- 本需求不与现有 ADR 冲突：它保持单一 Agent Runtime Backend、稳定的 Device Chat Contract 和统一 WebSocket 入口。
