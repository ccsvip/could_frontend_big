# ASR 忽略词条与设备有效输入超时

Status: ready-for-agent

## Problem Statement

设备的 Agent 语音交互会把上游 ASR 返回的纯标点（例如单独一个“。”）当作有效文本，也无法区分噪音被误识别成“对。”与用户真实说出“对”的情况。纯标点可能被显示、累计并触发 LLM；“对”这类可能有业务语义的短文本又不能安全地作为平台全局噪音词硬编码。

现有公司 ASR 语气词页面和后端只允许每行一个汉字，无法配置“好的”、“对对”、`um` 或 `you know` 等完整短文本。当实时 ASR 连续只产生噪音、纯标点或公司忽略词条时，Agent 链路会反复重开 ASR 监听，没有整轮等待上限，导致客户端长期处于请求或录音状态。

## Solution

将公司“语气词词表”扩展为 **Company ASR Ignored Transcript Set（公司 ASR 忽略词条集）**：每行一条完整短文本，支持单字、多字中文、拉丁字母、数字、内部空格与内部标点。保存与运行时使用同一规范化规则做完整文本精确匹配，不做包含匹配或模糊匹配。纯标点、纯符号和空白不属于公司配置，由平台在 preview 与 final 阶段统一判定为无效 ASR 并无条件过滤。

在 `/ai-models/asr` 新增独立的“设备语音交互”区块，允许当前公司配置 **ASR Effective Input Timeout（ASR 有效输入等待上限）**。取值为 5–120 秒的整数，公司未设置覆盖值时使用固定的平台默认值 15 秒，并可随时恢复默认。该上限只影响设备/Agent 自动语音交互，不影响管理页人工 ASR 测试，也不替代 VAD 静音时长。

计时从上游 ASR 准备完成开始；噪音、无效转写和公司忽略词条不重置计时。首个未被过滤的有效 preview 立即取消整轮超时，使长发言仍由 VAD 和现有断句逻辑正常完成。若到期仍无有效输入，将其作为正常的无输入结束，停止 ASR 和 Agent 本轮，不调用 LLM/TTS，也不发送错误事件。

## User Stories

1. 作为设备使用者，我希望单独的句号不会显示或触发回答，以便噪音不会变成伪造的问题。
2. 作为设备使用者，我希望纯标点、纯符号和空白转写被系统自动忽略，以便不需要管理员重复配置无语义内容。
3. 作为设备使用者，我希望被过滤的内容不会短暂出现在打字机预览中，以便界面不会闪现“。”或忽略词。
4. 作为设备使用者，我希望说出有效句子后立即看到实时预览，以便 ASR 打字机效果保持流畅。
5. 作为设备使用者，我希望有效长发言不会在第 15 秒被强制截断，以便可以完整说完问题。
6. 作为设备使用者，我希望连续只有噪音或忽略词时会话能有界地结束，以便设备不会永久录音或等待。
7. 作为设备使用者，我希望无有效输入超时不会触发 LLM 或 TTS，以便不会为噪音生成回答。
8. 作为设备运行时客户端，我希望超时时先收到输入停止事件，以便立即停止麦克风、PCM 上送和录音 UI。
9. 作为设备运行时客户端，我希望随后收到 ASR 和 Agent 的正常完成事件，以便状态机可以回到可交互状态。
10. 作为公司管理员，我希望配置单字“对”为忽略词条，以便在确认类回答不重要的场景中抑制噪音误识别。
11. 作为公司管理员，我希望系统不会默认过滤“对”，以便需要真实确认回答的公司不会丢失用户输入。
12. 作为公司管理员，我希望配置“好的”、“对对”等多字中文短语，以便忽略词条与本公司的实际 ASR 结果一致。
13. 作为公司管理员，我希望配置 `um`、`uh` 和 `you know` 等拉丁字母词条，以便支持英文或混合语音环境。
14. 作为公司管理员，我希望 `Um`、`UM`、`um。` 和全角等价输入只保存为一条，以便词条集与真实匹配行为一致。
15. 作为公司管理员，我希望内部标点参与精确匹配，以便“那个，嗯”不会意外等于“那个嗯”。
16. 作为公司管理员，我希望纯标点词条被拒绝，以便页面不会混淆“平台无效 ASR”与“公司忽略词条”。
17. 作为公司管理员，我希望每条词条有明确长度限制和可定位错误，以便能快速修正不合法的行。
18. 作为公司管理员，我希望忽略词条集默认为空，以便升级不会引入未授权的文本过滤。
19. 作为公司管理员，我希望在独立的“设备语音交互”区块配置有效输入等待上限，以便不会把它与 VAD 断句参数混淆。
20. 作为公司管理员，我希望在 5–120 秒内灵活调整该上限，以便适配本公司的设备与现场噪声环境。
21. 作为公司管理员，我希望清除覆盖值后恢复平台默认 15 秒，以便不必手工记忆默认参数。
22. 作为公司用户，我希望本公司的忽略词条和超时设置不影响其他公司，以便运行策略保持完全隔离。
23. 作为平台管理员，我希望沿用现有 tenant scope 管理目标公司的设置，以便不新增越权查询通道。
24. 作为管理页用户，我希望人工 ASR 麦克风测试不受设备超时限制，以便可以手动控制测试时长。
25. 作为现有 API 消费者，我希望 `/filler-words/` 与 `fillerWords` 保持可用，以便字段扩展不破坏现有集成。
26. 作为平台维护者，我希望超时任务在会话取消、关闭或得到有效输入时被清理，以便不会向新会话发送过期事件。

## Implementation Decisions

- 使用领域词汇 Company ASR Ignored Transcript Set 和 ASR Effective Input Timeout；页面文案不再把多字、英文或数字条目称为“单字语气词”。
- 纯空白、纯标点和纯符号是平台级无效 ASR，无论公司是否有忽略词条都必须被抑制。
- preview 与 final 共用同一规范化和过滤边界；被过滤的 preview 不下发给客户端，不更新累计文本，也不取消有效输入超时。
- 保持现有 ASR 替换规则顺序：先对识别文本应用公司替换规则，再进行无效文本判定与忽略词条精确匹配。
- 规范化使用 Unicode 宽度兼容归一、英文大小写不敏感、首尾空白与首尾标点/符号移除、连续内部空白折叠为一个空格；内部标点保留并参与匹配。
- 每个规范化后的忽略词条最长 32 个 Unicode 字符，且必须至少包含一个 Unicode 字母或数字。无效行返回包含行号的可定位校验错误。
- 保存规范化后的唯一词条；等价条目自动合并，空行忽略。保留现有词条集总文本长度上限，不扩展为正则、包含匹配或模糊匹配。
- 现有公司的忽略词条集继续默认为空，不迁移、预置或推断“嗯”、“对”等默认词条。
- `/ai-models/asr` 将“语气词词表”改为“ASR 忽略词条”，同步更新说明、示例、校验反馈和非汉字禁止文案。
- 保持现有忽略词条 API 路径与 `fillerWords` 字段兼容；本期不重命名现有数据表、序列化内部字段或客户端 API 类型字段。
- 新增公司隔离的 ASR runtime settings REST 资源，使用复数名词路径 `/api/v1/ai-models/asr/runtime-settings/`，支持 GET 和 PATCH。
- runtime settings 返回当前有效的 `effectiveInputTimeoutSeconds` 及是否使用平台默认值；PATCH 接受 5–120 的整数覆盖值，并允许用 `null` 清除覆盖。
- 有效输入等待上限是公司运行策略，必须通过已有 tenant scope 解析；普通公司账号不得读写其他公司，平台管理员仅通过已有公司作用域管理目标公司。
- 平台默认有效输入等待上限固定为 15 秒，本期不提供另一个平台级管理界面。公司覆盖值为可空，空值表示继承平台默认。
- `/ai-models/asr` 新增独立的“设备语音交互”区块，不与平台单例 VAD 表单混合。页面显示当前有效值、是否继承默认，允许保存公司覆盖和“恢复默认”。
- 有效输入计时仅在 Device Agent 语音链路上启用，从上游 ASR 就绪后开始；管理页人工 ASR 测试、独立 ASR 会话和文本 Agent 会话不启用该计时。
- 首个经替换、规范化且未命中公司忽略词条的非空 preview 取消超时任务。后续由既有 VAD 静音时长和 continuation window 处理断句与多段合并。
- 超时任务必须绑定当前 Agent/ASR 会话世代，并在首个有效 preview、会话取消、客户端断开、正常完成或连接关闭时取消，避免过期任务终止新会话。
- 有效输入超时使用现有统一 WebSocket，不新增 URL。服务端依次发送 `asr.input_stopped`、`asr.done`、`agent.done`，三者均携带 `reason: "effective_input_timeout"` 及当前 `id`/`requestId`/`traceId`。
- 超时是正常的无输入完成，不发送 `agent.error`，不产生用户消息，不启动 LLM/TTS，不写入空的 Device Runtime Conversation 历史。
- 保持现有 Device Chat Contract、`X-Device-Code` 认证、统一 WebSocket 入口、ASR 打字机累计文本、VAD 断句、LLM/TTS 顺序和多公司隔离规则。

## Testing Decisions

- 测试只断言外部可观察契约：API 返回值、保存后的公司配置、WebSocket 事件和 LLM/TTS 是否启动；不断言内部辅助函数、定时器字段或任务对象实现。
- 第一个主要接缝是现有 Django API 测试，覆盖忽略词条与 runtime settings 两个公司资源。
- 忽略词条 API 覆盖单字、多字中文、拉丁字母、数字、内部空格和内部标点；覆盖 NFKC 宽度归一、casefold、内部空白折叠、外围标点移除和等价去重。
- 忽略词条 API 拒绝纯标点/符号、超过 32 字符的条目和不包含 Unicode 字母或数字的条目，并返回行号可定位错误。
- runtime settings API 覆盖未配置时的 15 秒有效值、5 与 120 边界、非整数与越界校验、公司覆盖、`null` 恢复默认、普通公司越权防护及平台管理员 tenant scope。
- 第二个主要接缝是现有统一 WebSocket ASGI 集成测试，从 Agent 语音会话开始驱动真实的过滤、累计、完成和超时边界。
- WebSocket 回归覆盖纯句号 preview/final 不下发且不调用 LLM，公司忽略词 preview/final 不下发，内含更多有效文本的 preview 恢复下发并取消超时。
- WebSocket 超时回归使用可控时钟或极短测试配置，断言 `asr.input_stopped`、`asr.done`、`agent.done` 的严格顺序、原因与追踪字段，且断言不存在 ASR 转写、LLM/TTS 或空历史记录。
- WebSocket 竞态回归覆盖超时前到达的有效 preview、主动取消、客户端断开、上游正常完成和紧接着开始的新会话，确保过期超时不泄漏。
- 回归验证管理页人工 ASR 测试不会在公司超时到期时自动结束，现有 VAD 范围、替换词规则和独立 ASR 会话契约保持不变。
- 前端不新建低层单元测试接缝；使用生产构建验证 TypeScript、API 类型、页面组件和响应式布局集成。
- 后端测试在 Docker Compose 环境运行目标 Django 测试并优先使用 `--keepdb`；前端在 `web/` 下运行生产构建。

## Out of Scope

- 不通过文本猜测“对”是真实回答还是噪音误识别；是否忽略由公司显式配置。
- 不引入 ASR 置信度、音频质量评分、降噪、回声消除、客户端本地 VAD 或新 ASR 供应商。
- 不支持正则表达式、包含匹配、语义匹配或模糊匹配。
- 不为任何公司预置“嗯”、“啊”、“哦”、“对”或其他忽略词条。
- 不增加平台默认超时的可配置界面；平台默认值固定为 15 秒。
- 不改变 VAD 阈值、VAD 静音时长、continuation window 或上游 ASR 断句契约。
- 不将有效输入超时应用到管理页人工 ASR 测试、独立 ASR WebSocket 会话或文本 Agent 会话。
- 不重命名现有 `/filler-words/` API、`fillerWords` 字段、数据表或内部模型；页面与领域文档使用新词汇。
- 不新增第二条 WebSocket URL，不改变设备码认证、Agent Runtime Backend 选择、第三方会话 ID 或 LLM/TTS 内容契约。

## Further Notes

- 本规格在忽略词条字符范围、长度、规范化和过滤边界上取代旧“公司 ASR 语气词词表”规格中“只允许单个汉字”的约束；现有公司隔离和精确整文匹配原则仍保留。
- 领域定义记录在 `CONTEXT.md`；公司运行时设置不应复用平台单例 ASRConfig 的取舍记录在 ADR-0005。
- 实施必须同步更新设备实时契约文档，说明 `reason: "effective_input_timeout"` 的正常完成语义。
