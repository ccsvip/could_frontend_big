# 设备应用删除功能

Status: ready-for-agent

## Problem Statement

设备管理页面的“应用”标签（`/devices?tab=applications`）支持创建和编辑 Device Application（设备应用），但不能删除已创建的设备应用。管理员无法清理废弃配置，且现有设备和授权码与设备应用存在关联，直接删除必须避免误删授权码、破坏设备运行时配置或跨公司影响数据。

## Solution

在设备应用列表中提供受权限控制的删除操作。用户发起删除时，系统先计算并展示实际受影响的设备与授权码数量；确认后，删除设备应用、解除受影响设备的设备应用绑定，并把关联授权码保留为未分配授权码。

未分配授权码保留其授权码、授权类型、到期时间、使用记录及备注，但不能用于新的设备激活。管理员需要在设备授权中心为未分配且可编辑的授权码重新指定设备应用，之后该授权码才能重新用于激活流程。

## User Stories

1. 作为公司管理员，我希望在设备应用列表中删除废弃的设备应用，以便清理不再使用的设备配置。
2. 作为公司员工，我希望在已有设备管理权限的范围内删除设备应用，以便维护公司设备配置。
3. 作为没有设备删除权限的用户，我希望看不到或无法执行删除操作，以便系统遵守既有权限模型。
4. 作为管理员，我希望删除前看到将解除绑定的设备数量，以便评估删除对运行中设备的影响。
5. 作为管理员，我希望删除前看到将变为未分配状态的授权码数量，以便避免意外中断后续设备激活。
6. 作为管理员，我希望确认提示明确说明设备需要重新绑定应用，以便在删除后完成设备配置恢复。
7. 作为管理员，我希望确认提示明确说明授权码需要重新分配应用后才能再次用于激活，以便正确处理保留的授权码。
8. 作为管理员，我希望删除后受影响设备不再绑定已删除的设备应用，以便设备数据不引用不存在的配置。
9. 作为管理员，我希望删除后原有关联授权码仍保留，以便保留授权、到期和使用历史，而不是误删业务记录。
10. 作为管理员，我希望能够识别未分配授权码，以便处理因设备应用删除而待重新配置的授权码。
11. 作为管理员，我希望将未分配且可编辑的授权码重新分配给一个同公司的设备应用，以便恢复它的后续激活能力。
12. 作为管理员，我希望系统禁止使用未分配授权码激活新设备，以便避免设备在缺少运行时应用配置的状态下被授权。
13. 作为管理员，我希望已使用或已禁用授权码的历史状态保持正确，以便审计历史授权行为。
14. 作为平台管理员，我希望通过既有公司作用域管理当前公司设备应用，以便不改变现有多租户操作方式。
15. 作为普通公司用户，我希望只能查看和删除自己公司的设备应用及处理自己公司的授权码，以便保证公司数据隔离。
16. 作为运行时设备，我希望设备应用删除后收到或获取一致的完整运行时配置，以便不继续使用已删除应用的配置。
17. 作为运维人员，我希望删除失败时所有设备绑定和授权码关联都保持原状，以便不会出现部分删除的数据状态。

## Implementation Decisions

- 删除对象为 Device Application（设备应用），不涉及 Agent Application（智能体）及其会话、知识库或智能体运行时配置。
- 沿用设备资源的 REST 单资源删除接口：`DELETE /api/v1/devices/applications/{id}/`。不新增动词式删除或预览路径。
- 删除操作沿用既有 `DevicePermissionMixin` 的 destroy 权限、公司写权限和租户作用域；不新增独立权限点。跨公司查询、统计、删除和重新分配必须失败且不得泄露关联数量。
- 前端在设备管理“应用”标签的行操作中增加删除入口，并按已有权限能力控制可用性。
- 删除确认应使用删除前的服务端影响统计，显示实际受影响的直接绑定设备数和关联授权码数。统计及删除都必须受同一租户和权限约束，不能由前端缓存或跨页列表数据推断。
- 设备应用删除确认文案必须说明：受影响设备会解除设备应用绑定；受影响授权码会保留为未分配授权码；设备需重新绑定应用，授权码需重新分配设备应用后才能再次激活。
- 删除提交必须在单个数据库事务内完成：锁定目标设备应用及相关记录，解除直接绑定设备的 `application`，将关联授权码设为未分配，再删除设备应用。任一步骤失败时整体回滚。
- 授权码与设备应用的关系改为允许为空。删除不再级联删除授权码；未分配授权码保留授权码值、状态、授权类型、到期时间、使用设备、使用时间、创建人、备注和审计语义。
- 授权中心需将未分配作为明确可见状态，并支持按分配状态筛选。未分配且未使用、未禁用的授权码可重新分配给同公司设备应用；已使用或已禁用授权码保留历史，不应借重新分配改变既有使用或禁用事实。
- 创建或更新授权码时，绑定的设备应用必须属于当前租户；重新分配同样受此验证。未分配授权码不得通过设备激活流程，并应返回可定位的业务错误，提示先分配设备应用。
- 设备应用删除成功后，对受影响在线设备按既有运行时配置机制发出完整配置刷新，而不是仅发送应用字段的增量。HTTP 运行时配置和统一 `/ws/realtime/` WebSocket 的后续拉取/推送结果均不得继续引用已删除应用。
- 删除结果返回标准无内容成功响应；前端成功后刷新设备应用列表、设备列表和影响到的授权中心数据。失败时显示服务端明确错误且保持确认弹窗可恢复操作。
- 既有授权日志和设备授权历史不删除；其可选应用引用在删除后可为空，用于保留历史可追溯性。

## Testing Decisions

- 测试以可观察业务行为为准，不断言内部 ORM 调用顺序或 UI 实现细节。
- 后端应覆盖：同公司有删除权限的用户删除设备应用成功；无删除权限被拒绝；跨公司访问、影响统计、删除和重新分配均被拒绝。
- 后端应覆盖：删除后直接绑定设备的应用为空；关联授权码仍存在且应用为空；设备应用及其多对多资源关联被删除；授权和设备历史记录保留。
- 后端应覆盖：未分配授权码不能用于新设备激活；将未分配且未使用授权码分配给同公司设备应用后可恢复正常激活；已使用和已禁用授权码的状态不会被重新分配绕过。
- 后端应覆盖：删除过程任一失败会回滚设备解绑、授权码未分配和设备应用删除，避免部分状态。
- 运行时测试应覆盖：删除影响到的设备通过 `X-Device-Code` 获取完整运行时配置时不再携带已删除设备应用；统一 WebSocket 配置通知不会破坏既有订阅、在线状态或租户过滤。
- 前端应覆盖：有删除能力时显示删除操作；点击后展示服务端返回的实际影响数量和风险说明；确认成功后刷新列表；取消和失败不移除当前列表记录。
- 前端应覆盖：未分配授权码在授权中心可见、可筛选，并只在允许的状态和同公司目标应用下可重新分配。
- 后端测试优先复用现有设备授权 API、设备运行时配置和跨租户隔离测试的 fixture 与测试边界；前端验证至少执行 `npm run build`。

## Out of Scope

- 删除或修改 Agent Application（智能体）、智能体会话、知识文档或智能体配置。
- 删除设备本体、设备组、音色、资源库资源、模型资产、指令分组或滚动文本。
- 修改授权类型、到期规则、软件试用展示规则或已经完成的设备授权记录。
- 引入额外 WebSocket URL、额外设备认证方式或绕过既有 `X-Device-Code` 约定。
- 通过跨公司 query 参数绕过现有租户作用域。

## Further Notes

- 本规格的决策来源为 `.scratch/device-application-delete/map.md` 及其已解决票据。
- 当前实现中设备绑定删除已具备 `SET_NULL` 语义，而授权码关联仍为必填级联关系；实施需要数据库迁移与序列化/激活校验同步变更，不能只增加前端删除按钮。
- 删除与影响统计之间可能出现并发变动；确认提交时后端应以事务内的实时关联为准执行，并让前端以最终结果刷新为准。
