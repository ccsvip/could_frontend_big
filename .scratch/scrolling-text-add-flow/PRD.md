# 滚动文本连续新增与插入体验

Status: ready-for-agent

## Problem Statement

管理用户在滚动文本编辑弹窗中维护多条中英文文本时，新增条目会持续追加到内容末尾，而“新增文本”入口位于随内容一起滚动的顶部。条目较多时，用户每新增一条都需要先向上滚动找回入口，再向下滚动找到新条目并点击中文输入框，连续录入效率低且容易中断。

“向上插入”也会创建新的滚动文本条目，但目前没有统一的定位和聚焦反馈，用户无法立即确认插入位置并继续录入。

## Solution

滚动文本编辑弹窗保留顶部的“新增文本”入口，并使该入口在条目列表滚动时固定可见。国际化方案和新增入口保持在表单上方，条目卡片改为独立的响应式滚动区域。

无论用户追加新条目还是在现有条目上方插入，都自动把条目列表滚动到新卡片，并将输入焦点置于新条目的中文文本输入框。用户可以不离开当前操作流，连续完成多条中英文滚动文本录入。

## User Stories

1. As a 管理用户, I want 在编辑滚动文本时始终看到“新增文本”入口, so that 我无需反复向上滚动才能继续添加内容。
2. As a 管理用户, I want 国际化方案选择与新增入口保持在表单顶部, so that 我能在录入时持续确认当前文本配置。
3. As a 管理用户, I want 长文本列表只在条目区域内滚动, so that 弹窗标题、表单操作和确认按钮不会被大量条目推离可视区域。
4. As a 管理用户, I want 点击“新增文本”后立即看到新增的最后一条卡片, so that 我知道操作已经成功且无需手动定位。
5. As a 管理用户, I want 新增条目的中文输入框自动获得焦点, so that 我可以直接开始输入而无需额外点击。
6. As a 管理用户, I want 输入中文后可以按现有方式继续填写英文, so that 中英文内容录入流程不被改变。
7. As a 管理用户, I want 在某条文本上方点击“向上插入”后立即看到插入的新条目, so that 我能确认其顺序正确。
8. As a 管理用户, I want 向上插入的新条目的中文输入框自动获得焦点, so that 追加和插入操作拥有一致的连续录入体验。
9. As a 管理用户, I want 新增和向上插入后的滚动仅发生在条目列表内, so that 弹窗整体位置不会发生意外跳动。
10. As a 管理用户, I want 删除现有条目后保持当前位置和现有表单内容, so that 删除操作不会引入与新增无关的焦点或滚动干扰。
11. As a 管理用户, I want 在只有一条文本时继续受到现有最少保留一条的限制, so that 后端要求至少配置一条滚动文本的规则不被破坏。
12. As a 管理用户, I want 新增和插入条目继续使用现有中文、英文必填校验, so that 保存时不会产生不完整的双语文本。
13. As a 管理用户, I want 在桌面和移动端都能访问固定的新增入口, so that 小屏设备不会因长列表出现横向溢出或无法连续操作。
14. As a 管理用户, I want 编辑已有滚动文本时继续看到原有条目顺序和内容, so that 本次体验改进不会改变已保存的业务数据。
15. As a 管理用户, I want 点击保存后继续沿用现有创建、更新和列表刷新结果, so that 交互优化不会影响滚动文本的持久化与展示。
16. As a 平台维护者, I want 仅在管理端表单内完成此次改进, so that 后端 REST API、设备运行时配置和租户隔离逻辑保持稳定。
17. As a 平台维护者, I want 追加和插入复用同一条“定位并聚焦新条目”交互规则, so that 后续维护不会出现两套行为不一致的实现。

## Implementation Decisions

- 本次只调整滚动文本管理页面中的编辑表单交互，不改变滚动文本、滚动文本条目、国际化方案或保存载荷的领域含义。
- 国际化方案选择和“新增文本”入口处于表单内容区顶部；新增入口在条目列表滚动期间保持 sticky 可见。
- 滚动文本条目卡片承载于独立的纵向滚动区域。该区域使用响应式高度约束，不使用导致移动端横向溢出的固定宽度。
- 追加操作在现有最后一条之后创建空白条目；创建完成后，条目区域滚动至该新条目，且新条目的中文输入框获得焦点。
- 向上插入操作在所选条目之前创建空白条目；创建完成后，条目区域滚动至插入位置，且插入条目的中文输入框获得焦点。
- 追加与向上插入必须复用一套“记录目标索引、等待动态条目提交、定位目标卡片、聚焦中文输入”的流程，避免按按钮分别维护滚动或焦点逻辑。
- 目标条目的定位只滚动独立条目区域，不滚动整个页面或 Modal 的外层内容。
- 聚焦必须发生在动态条目已渲染之后；当浏览器或组件实例暂不可用时，不得阻止新增、插入或既有表单校验流程。
- 删除条目不触发自动滚动或自动聚焦；现有“至少保留一条”和文本必填校验规则保持不变。
- 弹窗宽度、表单保存/取消语义、创建与更新请求、数据归一化、权限控制、列表刷新和预览功能均保持现有行为。
- 实现遵循现有 antd 组件、品牌 token、流体排版和响应式规范；不新增图标库、不引入硬编码主色、不使用 Tailwind `!` 覆盖。
- 不新增 HTTP API、数据模型、数据库迁移、WebSocket 事件或设备运行时配置字段。

## Testing Decisions

- 主要验证缝为滚动文本管理表单的用户可观察行为：点击追加或向上插入后，目标新条目存在于正确顺序、位于条目滚动区域可见位置，且中文输入框成为活动焦点。
- 若项目现有前端测试环境支持页面交互测试，应覆盖创建态与编辑态各一次追加，以及编辑态一次向上插入；断言用户可见结果，不断言 ref、effect、DOM 查询或内部状态变量。
- 如现有前端尚无适合的自动化交互测试缝，至少通过本地运行后的浏览器验证覆盖上述三个场景，并记录其为后续前端测试基础设施的缺口；不为本次小范围体验修复引入新的测试框架。
- 补充长列表场景：多条卡片超过可视高度时，顶部新增入口仍可操作，追加后仅条目区域移动到末尾，表单外层不发生意外滚动。
- 补充移动端视口场景：条目区域可纵向滚动，新增入口可访问，页面与弹窗不出现横向溢出。
- 回归保存行为：新增空白条目时沿用中文和英文必填提示；填写完成后，创建或更新请求继续按现有顺序提交条目。
- 回归删除行为：删除不会触发新增式滚动或焦点跳转，且最后一条不可删除的现有规则继续生效。
- 前端改动完成后至少执行现有前端生产构建，以验证 TypeScript 与打包集成。

## Out of Scope

- 修改滚动文本的后端 REST API、序列化字段、模型、数据库迁移或数据校验。
- 修改设备运行时配置、设备 HTTP 认证、统一 WebSocket 协议或滚动文本在设备端的展示方式。
- 增加拖拽排序、批量导入、模板复制、文本搜索或自动翻译能力。
- 修改滚动文本标题的自动生成规则、启用状态、国际化方案种类或中英文必填约束。
- 改造其他资源管理页面的动态表单交互。
