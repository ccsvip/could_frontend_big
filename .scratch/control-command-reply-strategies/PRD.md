# 控制指令执行回复策略
Status: ready-for-agent

## Problem Statement

Company User 已能为公司维护并执行控制指令。控制指令被本地快速分发后，当前默认使用固定的“已执行：{指令名称}。”作为反馈，速度稳定但在指令名称本身是用户口语时不够自然；另一条已有路径会通过上游 LLM 根据执行结果生成适合语音播报的中文，文案自然但会增加等待时间，并可能受到超时、503 等上游故障影响。

Company User 需要为每条控制指令自行维护清晰、适合 Device TTS Runtime Audio 播放的执行回复。当没有维护回复时，管理员还需要能逐条选择固定反馈或智能生成反馈，并让 Device Runtime Conversation 始终得到明确、可播报的结果。

## Solution

为每条 Company-scoped Control Command 增加可选的“执行回复”和“未填写时回复方式”。已填写执行回复时，系统直接使用该文本；未填写时，管理员逐条选择固定反馈“已执行：{控制指令名称}。”或智能生成反馈。

固定反馈不调用上游 LLM。智能生成反馈在控制指令已下发后才使用已有自然中文播报能力生成回复，不重新进行 Tool Calling 或指令选择；生成不可用时自动回退固定反馈。所有路径继续通过既有统一 WebSocket 的 Device Chat Contract 返回 `llm.delta`、TTS 片段、`llm.done` 与 `agent.done`。

## User Stories

1. 作为 Company User，我希望为一条控制指令填写执行回复，以便设备完成下发后向现场用户播报我指定的文本。

2. 作为 Company User，我希望不填写执行回复时仍能为该控制指令选择回复方式，以便在速度与自然表达之间自行取舍。

3. 作为 Company User，我希望固定回复明确包含本次执行的控制指令名称，以便用户知道系统处理了什么动作。

4. 作为 Company User，我希望选择智能生成回复后，控制指令先被下发，再生成适合语音播报的自然中文，以便不会把文案生成误当成指令识别或执行的前置条件。

5. 作为 Company User，我希望智能生成服务超时、503 或未配置时仍收到固定的清晰反馈，以便不会得到空回复、隐私拒答或无法定位的错误。

6. 作为 Device User，我希望自定义执行回复与固定回复都能进入现有 TTS 队列，以便文本显示和设备播报保持一致。

7. 作为 Device User，我希望控制指令下发失败时收到失败结果，而不是成功执行回复，以便能够判断是否需要重试或检查设备。

8. 作为 Company User，我希望只能查看和编辑本公司的控制指令回复配置，以便保持公司数据隔离。

9. 作为平台管理员，我希望按既有 tenant scope 浏览公司控制指令时看到该公司的回复配置，以便能够支持配置排查而不绕过租户边界。

10. 作为 Company User，我希望已有控制指令在升级后保持可用，并默认采用固定回复，以便无需批量补填才能继续使用。

11. 作为 Company User，我希望控制指令管理页面明确区分“执行回复”和“未填写时回复方式”，以便知道哪项配置实际会生效。

12. 作为平台维护者，我希望 `llm.done` 中可观察本次回复来自自定义文本、固定规则、智能生成或智能生成回退，以便现场排查延迟和上游故障。

13. 作为平台维护者，我希望固定和自定义回复不增加上游 LLM 调用，以便本地高置信控制指令继续保持快速响应。

14. 作为平台维护者，我希望智能生成只接收本次已执行控制指令及其下发结果，不重新携带全量控制指令到上游，以便不扩大请求体、延迟或租户暴露面。

## Implementation Decisions

- 本需求仅覆盖 Control Command。Task Command、Task Command Step、任务执行和任务管理页面不新增回复字段或回复策略。

- 每条 Control Command 保存两个公司隔离字段：可为空的执行回复文本，以及未填写执行回复时的回复策略。回复策略只有 `fixed` 与 `generated` 两种稳定取值；新建和历史控制指令均默认 `fixed`。

- 生效优先级固定为：非空执行回复优先；否则按回复策略处理。保存时去除首尾空白，空白文本视为未填写。

- `fixed` 的成功回复固定为“已执行：{控制指令名称}。”；执行器返回失败、超时或网络错误时继续使用已有失败结果，不得播放成功回复。

- `generated` 仅在控制指令下发完成且执行器返回成功后调用一次平台 LLM，沿用已有“根据指令执行结果生成一句简短、适合直接转成语音播报的中文”的语义。它不得重新进行 Tool Calling、候选匹配或命令下发。

- 当 `generated` 的模型配置缺失、上游调用失败、超时、503 或未产生有效文本时，回复必须回退到该控制指令的 `fixed` 成功回复；控制指令的下发结果不能因文案生成失败被改写为普通 Agent Application 对话或安全拒答。

- 控制指令的快速本地分发继续保持异步数据库边界和异步 UDP/TCP 下发。使用自定义或固定回复时，不产生任何上游 LLM 请求；选择 `generated` 时，设备动作先执行，随后才等待一轮仅生成反馈的 LLM 调用。

- 现有歧义候选的有限 Tool Calling 路径在选定控制指令后，也必须按该控制指令的回复配置决定最终播报。它不得因为回复策略而重新发送全量控制指令。

- 现有统一 WebSocket URL、`agent.session.start`、`llm.delta`、Device TTS Runtime Audio、`llm.done`、`agent.done` 的顺序和 Device Chat Contract 保持兼容。`llm.done.commandDispatch` 在保留分发路径信息的同时，增加稳定的回复来源标记：`custom`、`fixed`、`generated` 或 `generated_fallback`。

- Control Command 的 REST Serializer、控制指令列表和详情响应、创建和局部更新请求使用既有前端 camelCase 约定暴露执行回复与回复策略。权限继续通过现有控制指令读写权限与 tenant scope 集中处理。

- 控制指令管理界面的新增与编辑表单增加“执行回复”多行输入和“未填写时回复方式”单选控件。执行回复非空时其优先级须在界面上清晰可见；编辑回填、创建默认值和列表展示应与 API 保持一致。

- 控制指令导入导出格式与前端 API 类型同步携带这两个字段；导入历史格式时缺少字段必须按空执行回复和 `fixed` 策略兼容，不得影响已有命令值、协议、IP、端口或分组。

- 运行时日志记录 tenant、控制指令标识、分发路径、回复来源、智能生成回退原因和执行结果；不得记录模型凭据或敏感配置。

## Testing Decisions

- 主测试缝为统一 WebSocket 的 `agent.session.start` ASGI 集成测试。这是 Device User 实际触发的最高层接口，应只断言事件、可播报文本、控制指令下发结果及回复来源，不依赖私有辅助函数或任务对象。

- 自定义执行回复场景：创建公司、设备、Agent Application 和控制指令，发送本地高置信文本；断言 UDP/TCP 下发、`llm.delta`、`llm.done`、`agent.done` 与自定义文本，且两个上游 LLM 入口均未被调用。

- 固定回复场景：控制指令未填写执行回复且策略为 `fixed`；断言“已执行：{控制指令名称}。”、`replySource=fixed`、TTS 片段和零上游 LLM 调用。

- 智能生成场景：控制指令未填写执行回复且策略为 `generated`；断言先完成一次控制指令下发，再调用一次仅用于自然反馈的 LLM，返回文本进入 Device Chat Contract，并记录 `replySource=generated`。

- 智能生成失败场景：模拟超时、503、空输出和模型配置缺失；断言控制指令已按原行为下发、结果退回固定回复、`replySource=generated_fallback`，且不会转为普通聊天或安全拒答。

- 控制指令下发失败场景：断言不使用自定义或固定成功回复，并通过既有失败契约反馈失败原因。

- 扩展现有控制指令 Serializer/API、指令工具、执行器和实时命令分发测试，覆盖租户隔离、默认策略、精确与轻微文本差异命中、有限候选 Tool Calling，以及多公司不可互见。

- 覆盖控制指令管理表单的新增、编辑回填和 API 类型；覆盖控制指令导出与旧格式导入的向后兼容。涉及 TypeScript 类型或页面逻辑时运行前端生产构建。

- 后端测试在 Docker Compose 的 `backend` 容器中使用 `--keepdb` 运行。已有实时 WebSocket、命令分发、命令工具和命令执行器测试作为回归保护。

## Out of Scope

- 不为 Task Command、Task Command Step 或任务管理页面增加执行回复或回复策略。

- 不增加第二条 WebSocket URL、动词式 HTTP API、设备侧认证方式或跨公司控制能力。

- 不调整控制指令的名称、命令码、协议、IP、端口、指令匹配阈值、有限候选策略或 UDP/TCP 执行语义。

- 不更换上游 LLM 供应商、模型、API Key、基础超时策略或 Third-Party Chatbot Interface 的运行时会话协议。

- 不将“已执行”升级为设备动作已被物理确认的承诺；当前执行成功仍以现有 UDP/TCP 下发结果为准。

- 不引入公司级全局回复模板、自动改写管理员自定义文本、批量 AI 生成回复或历史聊天记录回填。

## Further Notes

- 本规格建立在已规划的公司本地指令快速分发之上。自定义与固定回复是性能优先路径；智能生成是管理员逐条选择、明确接受额外等待的路径。

- “执行回复”面向 Device TTS Runtime Audio，应建议管理员填写简短、可直接播报的中文。系统不应为该文本再进行润色或调用 LLM。

- 自定义文本与回复策略均属于 Company-scoped Control Command 数据，必须遵循现有 Company User、平台管理员 tenant scope 和设备运行时的租户隔离边界。
