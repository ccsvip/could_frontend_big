# 智能体应用工作室设计

## 背景

当前系统已经有 `AgentApplication` 模型、`/api/v1/ai-models/applications/` 接口和 `/applications` 页面，可以创建一个带提示词、模型、知识库文档和调试对话的应用。但现有页面更像“应用管理表单”，没有形成类似 Dify 的“先创建应用卡片，再进入应用详情编排”的工作流；聊天链路也只使用系统提示词和历史消息，绑定的知识库文档尚未真正参与回答。

用户明确要求做 Solin 原生能力，不接入外部 Dify。目标是在现有能力上改造成一个能创建智能体的工作室：创建时填写名称和描述，进入后配置提示词、选择知识库、选择模型并调试预览。

## 目标

- 把现有应用入口改造成“智能体”工作台，位置推荐放在 `AI大模型 / 智能体` 二级菜单。
- 首次进入时展示卡片式创建入口；已有应用也以卡片展示。
- 点击创建后只填写应用名称和描述，创建成功后进入应用详情。
- 应用详情内提供侧边栏分区：`编排`、`日志与标注`、`监测`。
- `编排` 支持配置系统提示词、模型、知识库文档、temperature、max tokens、启用状态，并提供调试预览。
- 调试预览必须使用真实 `ChatConversation` / `ChatMessage`，方便后续在日志页查看。
- 知识库文档必须真正参与智能体回答生成。
- V1 知识检索仅支持 `.txt` 和 `.md` 文档；PDF、Office、表格等文件可以绑定和展示，但不参与检索。

## 非目标

- 不接入外部 Dify。
- 不做工作流画布、节点编排或多步骤 agent workflow。
- 不支持 DSL 导入。
- 不做变量系统。
- 不做发布、版本管理、灰度或多环境部署。
- 不新建 dataset、collection、vector store 等知识库集合模型。
- `日志与标注` V1 不做真实标注闭环，只读展示会话和消息。
- `监测` V1 不接入外部监控系统，只展示本地可计算统计。

## 信息架构

推荐将公司侧菜单从当前一级 `/applications` 调整为 `AI大模型 / 智能体`：

- 路由使用 `/ai-models/applications` 和 `/ai-models/applications/:applicationId`。
- 权限仍使用现有 `agent_applications.*` 权限点。
- 后端 API 继续使用 `/api/v1/ai-models/applications/`，避免无意义改动接口路径。
- 为兼容已有链接，前端可保留旧 `/applications` 路由重定向到新路径。
- 平台超管“公司视图”中同样把智能体放入 `AI大模型` 子菜单，而不是继续作为公司模块一级入口。

菜单迁移需要同步后端种子数据和相关测试，避免新公司仍获得旧的一级“应用管理”入口。

## 列表页

列表页是智能体工作室的第一个界面，不直接展示表单。

页面结构：

- 顶部为紧凑工具栏：标题、搜索、应用数量。
- 第一张卡片固定为“创建智能体”，点击打开创建弹窗。
- 已创建的智能体以卡片网格展示，每张卡片展示名称、描述、模型、知识库数量、启用状态、更新时间和进入按钮。
- 没有应用时仍展示“创建智能体”卡片，不使用大面积空状态替代主要操作。

创建弹窗只包含：

- 应用名称。
- 应用描述。

创建时后端自动设置：

- `temperature = 0.7`。
- `max_tokens = 1000`。
- `is_active = true`。
- 如果公司有可用默认 LLM 模型，则快照到 `llm_model`、`llm_provider`、`model_name`。

创建成功后立即跳转到详情页的 `编排` 分区。

## 详情页

详情页不再是左右两个大卡片的普通配置页，而是应用级工作区。

外层结构：

- 顶部栏展示返回、应用名称、模型状态、保存按钮。
- 左侧内部侧边栏展示 `编排`、`日志与标注`、`监测`。
- 主区域根据分区切换内容。

### 编排

`编排` 是 V1 的核心功能。

左侧配置区域：

- 基本信息：名称、描述、启用状态。
- 模型设置：从公司已授权且启用的 LLM 模型中选择。
- 提示词：系统提示词编辑器，支持多行文本。
- 知识库：多选现有 `KnowledgeDocument`。
- 生成参数：temperature、max tokens。

右侧调试预览：

- 使用当前应用创建或复用一个真实 `ChatConversation`。
- 发送消息走现有 SSE 流式聊天接口。
- 保存配置后，当前调试会话同步更新模型、提示词和生成参数。
- 调试消息写入真实 `ChatMessage`。

### 日志与标注

V1 为只读日志页，名字保留“标注”是为了后续扩展。

展示内容：

- 当前应用下的会话列表。
- 每条会话展示标题、摘要、消息数、最近更新时间、模型。
- 点击会话后展示消息明细。
- 消息明细展示用户消息、助手消息、创建时间和现有 `feedback` 字段。

V1 不提供：

- 人工改写答案。
- 标签体系。
- 训练集导出。
- 标注任务状态流转。

### 监测

V1 展示本地统计，不引入新的监控基础设施。

建议指标：

- 会话总数。
- 消息总数。
- 用户消息数。
- 助手消息数。
- 点赞数、点踩数。
- 最近 7 天会话数量趋势。
- 最近更新时间。

这些统计都从 `ChatConversation` 和 `ChatMessage` 聚合得到，按当前应用和公司作用域过滤。

## 知识库检索

V1 不引入向量数据库，采用可解释、低依赖的文本检索。

检索范围：

- 仅使用当前 `AgentApplication.knowledge_documents` 绑定的文档。
- 仅处理扩展名为 `.txt` 和 `.md` 的文档。
- 文档必须属于当前公司。
- 未审核通过或不可访问的文档不参与检索。
- PDF、Office、表格文件可以显示在已绑定列表中，但检索时跳过，并在 UI 中标明“暂不参与检索”。

检索方式：

- 后端读取文档文本内容，按段落或固定长度切片。
- 对用户当前问题做简单关键词匹配和计分。
- 取 Top N 片段注入到系统消息后、历史消息前的上下文中。
- 注入内容必须明确标识为“知识库参考”，并要求模型仅在相关时使用。

边界规则：

- 文本过长时限制最大注入字符数，避免挤占上下文。
- 检索失败不应中断聊天；应记录日志并降级为无知识库回答。
- 不把检索片段保存为额外 `ChatMessage`，避免污染用户可见对话。
- 可以在日志中记录命中的文档 ID 和片段数量，V1 不新增命中明细表。

## 后端改动

优先复用现有模型，避免新增表。

需要调整：

- `AgentApplicationViewSet.create`：创建应用时，如果请求没有传模型，自动使用当前公司的默认可用 LLM 模型。
- `ChatConversationViewSet.send`：当会话绑定了应用时，在构造 OpenAI-compatible `messages` 前执行知识库检索并注入参考上下文。
- 新增小型服务模块，例如 `apps.ai_models.services.agent_knowledge`，封装文档过滤、文本读取、切片、计分和 prompt 注入文本生成。
- `ChatConversationViewSet.get_queryset`：支持按 `application` 查询参数过滤当前应用会话，用于日志页。
- `AgentApplicationViewSet` 增加只读统计 action，例如 `GET /ai-models/applications/{id}/stats/`，用于监测页。

权限和租户边界：

- 应用、知识文档、会话、消息都必须按当前公司过滤。
- 平台超管按公司浏览时继续依赖现有 tenant scope。
- 公司管理员和公司员工除员工管理外权限一致，智能体功能不再单独区分。

## 前端改动

主要修改点：

- `web/src/views/application-management/index.tsx`
  - 保留为智能体工作室页面，但重构列表和详情结构。
  - 列表改为创建卡片 + 应用卡片。
  - 详情改为内部侧边栏分区。
  - 新增日志只读视图和监测视图。

- `web/src/api/modules/applications.ts`
  - 增加应用统计 API。
  - 增加应用会话列表 API 或在 `chat.ts` 中扩展会话列表查询。

- `web/src/api/modules/chat.ts`
  - 增加会话列表类型与 `applicationId` 查询参数。

- `web/src/router/index.tsx`
  - 增加 `/ai-models/applications` 路由。
  - 旧 `/applications` 路由重定向到新路由。
  - 平台超管公司视图同步增加 `/tenants/:tenantId/ai-models/applications`。

- `web/src/layouts/dashboard-layout.tsx`
  - 公司视图中将智能体移到 `AI大模型` 子菜单。
  - 激活菜单逻辑识别新旧智能体路径。

UI 约束：

- 业务后台应保持安静、密集、可扫描，不做营销型 hero。
- 卡片半径不超过现有业务风格，避免过度装饰。
- 列表和详情在桌面、平板、移动端都不能出现文字重叠或按钮溢出。

## 测试与验证

后端测试：

- 创建应用不传模型时，若公司有默认可用模型，应自动绑定该模型。
- 创建应用不传模型且公司没有可用模型时，应用仍可创建，但调试发送时返回明确错误。
- 应用只能绑定当前公司知识文档。
- 聊天发送时 `.txt` / `.md` 知识文档内容会进入模型请求消息。
- PDF / Office 文档绑定后不会参与检索，也不会导致发送失败。
- 会话列表按 `application` 过滤。
- stats action 只统计当前应用和当前公司范围内的数据。

前端验证：

- `docker compose exec web npm run build`。
- 列表页无应用时仍显示创建卡片。
- 创建弹窗只包含名称和描述。
- 创建成功后进入详情页。
- `编排` 保存配置后调试会话使用最新配置。
- `日志与标注` 能查看当前应用会话和消息。
- `监测` 展示当前应用统计。
- 超管公司视图菜单路径和普通公司用户菜单路径都能进入智能体。

如涉及后端改动，执行相关 Django 测试：

```bash
docker compose exec backend python manage.py test apps.ai_models.tests.test_agent_application_api
```

## 风险与处理

- 知识库文件读取路径可能受存储后端影响。实现时必须使用 `KnowledgeDocument.file` 的 Django storage API，而不是拼接本地路径。
- SSE 是异步流式链路，知识库检索不能在异步生成器中直接执行未包装的 ORM 查询。同步 ORM 操作需要在进入流式阶段前完成，或用 `sync_to_async` 包装。
- 简单关键词检索不等于语义检索。V1 目标是让知识文档真实参与回答，不承诺向量检索质量。
- Office/PDF 暂不检索容易造成用户误解，UI 必须清楚标注哪些文档参与检索。
- 菜单路径迁移会影响默认落地页和权限测试，需要同时更新种子迁移和前端超管公司视图。
