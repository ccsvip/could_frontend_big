# 修复图片资源批量删除未清理存储文件

## Goal

修复 `/api/v1/resources/images/bulk/` 批量删除图片资源时只删除数据库记录、未删除实际上传文件的问题，避免上传目录或对象存储持续残留无主文件。

## Background

- 图片资源批量删除复用 `ImageResourceViewSet.perform_destroy()`，现有接口返回 `deletedIds` 与逐项 `failures`，采用部分成功语义。
- 删除前必须继续执行现有的租户隔离、图片类型限制、删除权限及智能体标注引用保护。
- 成功删除的资源可能使用 Django `FileField` 本地/配置存储，也可能使用 `objectKey` 指向 local/R2 对象存储。
- 当前回归测试只断言数据库记录消失，没有断言底层文件或存储对象被清理，因此未覆盖用户报告的症状。

## Requirements

- 批量删除中每个成功删除的图片资源必须同步清理其实际存储内容。
- 对带有上传文件的资源，删除前保存文件名与存储实例，并在数据库删除成功后通过该存储实例删除文件。
- 对带有 `objectKey` 的资源，继续通过现有 `delete_object(objectKey, backend=storageBackend)` 路径清理对象。
- 同一资源不得因字段组合而重复删除同一个底层对象。
- 未找到、越权、类型不符或被引用保护的资源不得删除数据库记录或底层文件。
- 保持现有 REST 路径、请求体、响应结构、部分成功顺序及权限行为不变。
- 存储清理实现应位于单项和批量删除共用的删除边界，避免两条路径产生不同语义。

## Acceptance Criteria

- [ ] 上传到测试文件存储的图片经批量删除后，数据库记录不存在，实际文件也不存在。
- [ ] 带 `objectKey` 的图片经批量删除后，按该资源记录的 `storageBackend` 调用一次对象删除。
- [ ] 被引用保护的图片批量删除失败时，其数据库记录和实际文件都保持不变。
- [ ] 批量删除 API 的 `deletedIds`、`failures`、租户隔离和权限契约保持不变。
- [ ] 新增能够直接捕获“数据库已删但文件仍存在”的回归测试，并先验证修复前失败、修复后通过。
- [ ] 相关 Django 资源 API 测试通过。

## Out of Scope

- 清理历史上已经产生的无主文件。
- 修改 `/resources/images` 前端批量管理交互或 API 契约。
- 引入异步清理任务、回收站或存储迁移机制。
