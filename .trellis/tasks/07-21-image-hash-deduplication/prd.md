# 图片 Hash 去重上传

## Goal

为 `/resources/images` 图片资源上传建立基于文件内容的去重机制，记录图片内容摘要，并阻止同一公司重复保存相同图片，减少重复资源与对象存储占用。

## Background

- 图片资源属于 tenant，所有查询与写入必须保持公司数据 100% 隔离。
- 普通存储路径通过 Django multipart 接收图片；R2 路径由浏览器先直传对象存储，再调用图片资源创建接口保存 `objectKey`。
- 批量上传在普通存储下使用 `POST /api/v1/resources/images/bulk/`；R2 下当前逐张直传并逐张创建资源。
- `Resource` 当前没有文件内容摘要字段，也没有图片内容唯一约束。

## Requirements

- 对图片文件使用 SHA-256 计算稳定的内容摘要，并在资源记录中保存摘要。
- 重复判定范围限定为同一 tenant 的图片资源；不同 tenant 上传相同内容互不影响，也不得暴露对方资源信息。
- 后端必须执行最终重复校验，前端计算仅用于在上传文件字节前提前拦截，不能作为唯一防线。
- 单图上传、普通批量上传和 R2 直传图片三条路径必须遵循相同去重规则。
- 同一请求内选择内容相同但文件名不同的图片，也应识别为重复内容。
- 并发请求不得创建同一 tenant 下相同摘要的两条图片资源记录。
- 重复错误必须明确指出图片已存在，不返回其他 tenant 的资源信息。
- 批量上传采用部分成功策略：跳过重复图片，继续创建其余新图片，并返回被跳过的文件名及原因。
- 上线前已有图片通过可重复执行的后台/管理命令读取本地文件或对象存储并回填摘要；数据库迁移不得访问对象存储。
- 历史回填发现重复内容时保留所有资源，不自动删除或改写引用；最早创建的记录作为去重基准，其余重复记录保持未索引状态并写入回填报告。
- 视频资源不在本次去重范围内。

## Acceptance Criteria

- [ ] 首次上传图片成功后，资源记录保存 64 位 SHA-256 摘要。
- [ ] 同一 tenant 再次上传相同内容时不会创建新的资源记录，也不会保留新的对象存储文件。
- [ ] 同一 tenant 上传同内容不同文件名时仍判定重复。
- [ ] 不同 tenant 可分别上传相同内容，且响应不泄露跨 tenant 信息。
- [ ] 单图、普通批量和 R2 图片直传均覆盖重复校验。
- [ ] 批量上传混有重复项时，新图片正常创建，重复项不创建且前端明确列出跳过项。
- [ ] 并发上传由数据库约束保证同 tenant 图片摘要唯一。
- [ ] 历史图片 Hash 回填命令可重复执行，并兼容本地文件与 R2/MinIO 对象。
- [ ] 历史重复图片不会被自动删除；回填结果列出基准资源和保留的重复资源。
- [ ] 相关 Django 测试、迁移检查和前端构建通过。

## Out Of Scope

- 视频资源去重。
- 跨 tenant 共享同一个对象存储对象。
- 感知哈希或视觉相似图片检测；仅按文件字节完全一致判断。
