# 唤醒词通知增强方案

## 背景

用户反馈：唤醒词修改后虽然有通知，但安卓仍需要重启才生效；前端/安卓同事认为如果通知内容更详细，就可以立即生效。

已确认现状：

- 后端已有事件 `device.wake_words.changed`，由 `backend/apps/devices/views.py` 的 `WakeWordViewSet._publish_wake_word_changed()` 发布。
- 事件会通过统一 WebSocket `/ws/realtime/` 下发，外层为 `devices.event`。
- 安卓可通过 `device.runtime_config.subscribe` 或 `device.status.start` 订阅设备级事件。
- 当前事件 payload 主要是配置失效提示：`action`、`wakeWordId`、`text`、`deviceCodes`、`refresh.endpoint`。
- 安卓完整运行时配置来自 `GET /api/v1/device-runtime/config/`，唤醒词字段由 `DeviceRuntimeConfigView._wake_words_payload()` 返回：`wakeWords` 和 `wakeWordLines`。

## 根因判断

当前通知不是“完整唤醒词配置”，而是“唤醒词配置已变更，请刷新”的通知。

如果安卓收到事件后没有立即重新拉取 `/api/v1/device-runtime/config/`，或者安卓希望仅凭通知完成热更新，现有 payload 不够用，因为它没有给出设备当前完整可用的 `wakeWords` / `wakeWordLines`。这会导致本地唤醒词引擎仍使用旧配置，直到重启时重新加载配置才生效。

## 推荐方案

在保持现有事件类型、WebSocket 入口、刷新 endpoint 不变的前提下，增强 `device.wake_words.changed` payload，直接携带目标设备当前完整唤醒词运行时配置。

推荐新增字段：

```json
{
  "type": "device.wake_words.changed",
  "action": "create|update|delete",
  "resource": "wakeWords",
  "tenantId": 1,
  "wakeWordId": 123,
  "text": "你好小德",
  "deviceIds": [10],
  "deviceCodes": ["ANDROID-001"],
  "wakeWords": [
    {
      "id": 123,
      "text": "你好小德",
      "encodedText": "n ǐ h ǎo x iǎo d é",
      "keywordLine": "n ǐ h ǎo x iǎo d é @你好小德 :2.0 #0.25",
      "boost": 2.0,
      "threshold": 0.25
    }
  ],
  "wakeWordLines": [
    "n ǐ h ǎo x iǎo d é @你好小德 :2.0 #0.25"
  ],
  "refresh": {
    "endpoint": "/api/v1/device-runtime/config/",
    "reason": "wakeWordsChanged"
  }
}
```

字段含义：

- `wakeWords`：设备当前所有启用的唤醒词对象，结构复用 `DeviceRuntimeConfigView._wake_words_payload()`。
- `wakeWordLines`：设备当前所有启用的 sherpa-onnx keyword line，安卓可直接替换本地唤醒词配置。
- `refresh`：继续保留，作为兼容和兜底。安卓如果不能直接热更新，仍可按 endpoint 拉全量配置。

## 实现步骤

1. 在 `backend/apps/devices/views.py` 增加一个小的设备唤醒词 payload 组装逻辑。

   - 复用 `DeviceRuntimeConfigView._wake_words_payload(device)`，避免两个地方字段不一致。
   - 根据 `device_ids` 查询目标设备，按设备当前绑定关系生成唤醒词配置。
   - 当前唤醒词绑定逻辑是一个事件可包含多个设备，实际 `wakeWords` / `wakeWordLines` 在不同设备上可能不同；为避免歧义，推荐新增 `runtimeConfigByDeviceCode`。

2. 调整事件 payload，兼顾单设备和多设备。

   推荐结构：

   ```json
   {
     "runtimeConfigByDeviceCode": {
       "ANDROID-001": {
         "wakeWords": [],
         "wakeWordLines": []
       },
       "ANDROID-002": {
         "wakeWords": [],
         "wakeWordLines": []
       }
     }
   }
   ```

   为方便安卓处理，如果 `deviceCodes` 只有一个，也可同时在顶层放 `wakeWords` 和 `wakeWordLines`，其值等于该设备配置。

3. 处理 `update` 的旧设备和新设备。

   - 当前 `perform_update()` 已经合并旧绑定设备和新绑定设备。
   - 对旧设备：更新后该唤醒词可能已不再绑定，payload 应返回旧设备“更新后的完整唤醒词列表”，通常会移除该词。
   - 对新设备：payload 返回新设备“更新后的完整唤醒词列表”，包含新绑定的词。
   - 因此必须在 `serializer.save()` 后，再用合并后的设备集合查询并生成完整配置。

4. 处理 `delete`。

   - 删除前保存 `device_ids` / `device_codes`。
   - 删除后为这些设备重新查询当前完整唤醒词配置。
   - 对被删词所在设备，`wakeWords` / `wakeWordLines` 应不再包含被删词。

5. 保持 WebSocket 和 REST API 不变。

   - 不新增 WebSocket URL。
   - 不新增动词式 HTTP API。
   - 不改变 `GET /api/v1/device-runtime/config/` 契约。
   - 不移除 `refresh.endpoint`，避免破坏现有安卓逻辑。

## 安卓消费建议

安卓收到：

```json
{
  "type": "devices.event",
  "payload": {
    "type": "device.wake_words.changed",
    "runtimeConfigByDeviceCode": {
      "当前设备码": {
        "wakeWords": [],
        "wakeWordLines": []
      }
    }
  }
}
```

处理顺序：

1. 取 `payload.runtimeConfigByDeviceCode[本机 deviceCode]`。
2. 如果存在，立即用其中 `wakeWordLines` 或 `wakeWords[].keywordLine` 替换本地唤醒词引擎配置，并重新加载唤醒词识别器。
3. 如果不存在或热更新失败，再按 `payload.refresh.endpoint` 请求 `/api/v1/device-runtime/config/`，请求头带 `X-Device-Code`。

   - 断言单设备时顶层 `wakeWords` / `wakeWordLines` 包含新词。
   - 断言 `runtimeConfigByDeviceCode[device.code]` 存在。

## 风险和边界

- 多设备事件中顶层 `wakeWords` 可能产生歧义，因此推荐 `runtimeConfigByDeviceCode` 作为权威字段。
- 通知 payload 会变大，但唤醒词数量通常很小，风险可控。
- 后端增强通知只能提供热更新所需数据；安卓仍必须在收到事件后主动替换本地识别器配置，否则仅靠收到通知不会自动生效。
