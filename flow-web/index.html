① 三个角色全景· ② 开通公司时序· ③ 三级菜单可见性· ④ 三道隔离防线· ⑤ 首登改密 / LLM 隔离· ⑥ 关键接口表
隔离地基

用 Tenant + Membership 两张表承载租户，刻意 不改 AUTH_USER_MODEL（仍是原生 auth.User，中途替换风险过高）。另有 12 张业务表加 nullable tenant 外键 + 按租户唯一约束，全部挂 TenantManager 走行级隔离。

三道防线

Manager（模型层 for_tenant，None fail-closed 返回空集）→ ViewSet Mixin（请求层自动收敛 + 强制注入 tenant）→ 缓存键（把 tenant_id 并入响应缓存键）。再加隔离契约测试防回归。

三类账号

平台超级管理员（运维，无 Membership）、公司管理员（审批通过那个人，is_tenant_admin=True）、公司员工（经 UserRole 绑定本公司角色）。菜单与权限按 Menu.audience 三态切分、层层收窄。

① 三个角色全景

点角色名切到「角色视角」，下方时序步骤、菜单矩阵、接口表会随之高亮。

平台超级管理员

运维

role payload → {code:'admin', name:'管理员'}

看到的菜单

所有 audience=allaudience=platform 的启用菜单，含 platform 专属的「租户管理」；明确排除 audience=tenant_admin 的「员工管理」。业务页对其返回 .all()，能看到全部公司数据（跨租户维护建议走 /admin/）。

能力
  • 持有全部启用权限点（含 tenant.management.view），通过 CanManageTenants 把关的平台端点。
  • GET/POST/PATCH/PUT /tenants/ 管理所有公司（无 DELETE，停用走 is_active）。
  • GET/PUT /tenants/<id>/menus/ 给公司分配可见菜单 + 权限点，设定其可见范围上界。
  • GET /menus/catalog/ 拿可分配目录（仅 audience=all 菜单 + 权限点）。
  • 审批账号申请：GET/PATCH /auth/account-applications/manage/<pk>/。
  • 无 Membership，视为平台运维；JWT 不写 tenant_id。

公司管理员

即申请人

role payload → {code:'tenant_admin', name:'公司管理员'}

看到的菜单

超管分配给本公司的 audience=all 菜单（Tenant.menus）＋ 全部 audience=tenant_admin 菜单（如「员工管理」，固有可见、不依赖分配）。看不到 platform 专属的「租户管理」。

能力
  • 权限 = Tenant.permission_points ＋ 固有的 tenant.employees.manage（建号能力不依赖授权）。
  • GET/POST/PATCH /employees/ 管理本公司非管理员成员 ＋ POST /employees/<id>/reset-password/。
  • GET/POST/PATCH/DELETE /roles/ 管理本公司角色；菜单/权限点服务端钳制在 Tenant 授权范围内，越界 400；有员工绑定的角色不可删。
  • GET /my-tenant/catalog/ 拿本公司视角可分配目录（Tenant.menus[all] + Tenant.permission_points）。
  • 建员工时置 must_change_password=True，员工首登强制改密。
  • 本人由 provision_company 设为 is_tenant_admin=True。

公司员工

受双重裁剪

role payload → 所绑定角色的 {code, name}

看到的菜单

所绑定 Role.menus ∩ Tenant.menus（两侧都限 audience=all），永不超出公司被授权范围。未绑定角色则无菜单。

能力
  • 权限 = Role.permission_points ∩ Tenant.permission_points（交集裁剪）。
  • 经 UserRole 绑定本公司某个 Role（tenant 非空、is_template=False）。
  • Membership.is_tenant_admin=False，被公司管理员的 EmployeeViewSet 作用域纳管。
  • 首登被 must_change_password=True 拦在改密页，POST /auth/change-password/ 成功后解除。
  • 所有业务读写经三道防线收敛到本公司 tenant；无归属时 for_tenant(None) 返回空集。

① 提交开通申请匿名

申请人填 username / applicant_name / enterprise_name / phone / password(仅存哈希) / reason，status=pending。AllowAny。提交后触发 Celery notify_account_application.delay，不可用时同步兜底。

POST /api/v1/auth/account-applications/

② 超管查看待审申请超管

CanViewAccountApplications（account_applications.view）把关，列出 AccountApplication。

GET /api/v1/auth/account-applications/manage/

③ 审批通过 → 自动建公司超管

PATCH 改 status=approved。save 检测后：① ensure_login_user() 建 User（复制已哈希 password 防双重哈希）；② provision_company() 建 Tenant（generate_unique_tenant_code 唯一 slug，中文名回退 'company' 加序号）+ Membership(is_tenant_admin=True)，用 .update() 回写 tenant 避免递归 save。幂等：已有 Membership 复用现有公司。公司名取 enterprise_name or applicant_name or login_username。

PATCH /api/v1/auth/account-applications/manage/<pk>/

④ 给公司分配菜单与权限点超管

CanManageTenants（tenant.management.view）。PUT 写入 menuIds + permissionPointIds，设定公司可见范围上界（Tenant.menus / permission_points）。可先 GET /menus/catalog/ 拿可分配目录（仅 audience=all 菜单 + 权限点）。

PUT /api/v1/tenants/<id>/menus/

⑤ 公司管理员登录公司管理员

用申请时的用户名/密码登录。LoginView 用 get_user_tenant 解析公司，把 tenant_id 写入 JWT refresh 作辅助 claim；权威租户上下文始终以 DB Membership 为准。

POST /api/v1/auth/login/

⑥ 建角色（可选）公司管理员

CanManageEmployees（tenant.employees.manage）。TenantRoleSerializer 把 menuIds/permissionPointIds 服务端钳制在授权范围内，越界 400；role 自动 tenant=本公司、is_template=False；code 在公司内唯一（unique_role_code_per_tenant）。

POST /api/v1/roles/

⑦ 建员工公司管理员

CanManageEmployees。EmployeeCreateSerializer 校验 username 全局唯一（跨公司不可重名）+ 密码规则，建 User + Membership(is_tenant_admin=False, must_change_password=True)，可选 UserRole 绑定本公司角色（roleId 校验必须属于本公司）。

POST /api/v1/employees/

⑧ 员工首登强制改密员工

首登被前端 AuthGuard 依据 mustChangePassword 拦在改密页（该标志经 /me 与 EmployeeSerializer 暴露）。员工调 change-password 成功后，ChangePasswordView 清除本人 Membership.must_change_password 标志，解除拦截。

POST /api/v1/auth/change-password/

④ 三道隔离防线

行级隔离自下而上三层收敛，再加一层契约测试防回归。关键 fail-closed：无租户上下文时返回空集，而不是裸查泄漏。

第一道Manager · 模型层

tenant-scoped 模型挂 objects = TenantManager()（基于 TenantQuerySet）。业务路径走 .for_tenant(t)；关键 fail-closed：for_tenant(None) 返回 self.none() 空集，杜绝无租户上下文时裸查泄漏。标准 .all()/.filter() 保留给 admin/Celery/迁移。

backend/apps/tenants/managers.py

第二道ViewSet 基类 · 请求层

TenantScopedQuerysetMixin.get_queryset() 自动 .for_tenant(request_tenant)（superuser 旁路返回 .all()）；perform_create()/tenant_create_kwargs() 强制注入 tenant，前端伪造的 tenant 字段无效。租户由 services.get_request_tenant 从 request.user.membership 解析（无 TenantMiddleware，因 DRF JWT 鉴权在 view 层）；MRO 约定本 mixin 放 Cached 之后、PermissionMapped 之前。

backend/apps/tenants/mixins.py

第三道缓存键 · 缓存层

make_response_cache_key 把 tenant_id 并入响应缓存键（tenant 为 None 时用 'none'），sha256 摘要前带 t=<tenant_id>，避免 A 公司缓存的列表/详情被 B 公司请同一 URL 命中；DeviceViewSet.stats 统计键同样并入 tenant_id。

backend/config/business_cache.py

护栏隔离契约测试 · 防回归

tests/test_isolation_contract.py 在「带 tenant 外键但没挂 TenantManager」时 CI 报红；新增 tenant-scoped 模型必须挂 manager，或在 KNOWN_NON_SCOPED_EXEMPTIONS 登记豁免并写理由。

⑤ 员工首登强制改密 ＆ 每公司 LLM 隔离

员工首登强制改密

置位　由公司管理员在两处触发 Membership.must_change_password=True：① 建员工时 EmployeeCreateSerializer.create 固定置 True（employee_serializers.py）；② 重置密码时 EmployeeViewSet.reset_password 调 user.set_password 后 Membership...update(must_change_password=True)（employee_views.py）。

拦截　该标志经 /auth/me/ 与 EmployeeSerializer.mustChangePassword 暴露给前端，AuthGuard 据此把员工拦在改密页。

解除　员工调 POST /auth/change-password/，ChangePasswordView 先 set_password，再 ...update(must_change_password=False) 清标志，之后正常放行（accounts/views.py）。

超管/公司管理员本人不走此机制（建公司时 Membership 默认 must_change_password=False）。

每公司 LLM 供应商隔离

LLMProvider 是 tenant-scoped 模型：LLMProviderViewSet 继承 TenantScopedQuerysetMixin，列表/详情/写入全部按本公司收敛。聊天三处显式按租户限定 provider，绝不跨租户拿到别家公司的供应商（含其 API Key）：

  • ① 建会话 — provider_qs = LLMProvider.objects.for_tenant(tenant).filter(pk=provider_id)，会话本身写入 tenant=request_tenant。
  • ② update-config — for_tenant(conversation.tenant).filter(pk=provider_id, is_active=True)。
  • ③ send 兜底 — provider 缺失或停用时回退 for_tenant(conversation.tenant).filter(is_active=True).first()。

关键 fail-closed：for_tenant(None) 返回空集，无租户上下文时回退拿不到任何 provider，而非误取全局第一个。兜底仍取不到时返回 400「暂无可用的 LLM 供应商」并写入提示性助手消息，绝不静默跨租户取用。

附：数据模型一览（点击展开）

Tenant(name, code 唯一 slug, is_active, is_legacy 历史默认公司标记, menus M2M→accounts.Menu, permission_points M2M→accounts.PermissionPoint) — menus/permission_points 是超管分配给该公司的可见上界。

Membership(user OneToOne→auth.User, tenant FK, is_tenant_admin 标识公司管理员即审批通过那个人, must_change_password 首登强制改密) — superuser 不建 Membership，视为平台运维。

Role(name, code, tenant FK 可空, is_template, menus/permission_points M2M) — tenant=null 表平台模板(is_template=True)，非 null 表某公司自建角色，约束 unique(tenant, code) 使 code 在公司内唯一；员工经 UserRole(user OneToOne→Role) 绑定。

Menu 加 audience 字段（all/platform/tenant_admin）决定菜单归属与可分配性。

accounts.AccountApplication 加 tenant FK（SET_NULL）+ username/enterprise_name/password 哈希等字段，save() 在 approved 时建 User 并调 provision_company 回写 tenant。

另有 12 张业务表加 nullable tenant FK + 按租户唯一约束，全部挂 TenantManager 走行级隔离。
