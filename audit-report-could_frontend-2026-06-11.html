📝 总体评估

📋 总体评估

项目概况：solin 是一个 React 18 + Vite + TypeScript 前端、Django 5.2 + DRF + Celery + ASGI 后端的数字人后台管理平台。项目领域边界相对清楚，核心域包括账号权限、租户隔离、设备运行时、资源/指令、AI/ASR/聊天、知识库和审计日志；根 compose 能把 db、redis、backend、celery worker/beat、web 串起来。

主要风险：当前不适合直接作为稳定公开发布包落地，最大阻塞是安全与发布配置：compose 无条件创建已知超管账号，当前运行时加载 `config.settings.base` 而不是 `prod`，`docker compose config` 会展开 `.env` 密钥，且 `db` 容器获得了不需要的业务密钥。其次是设备激活并发可产生重复设备、前端依赖存在 high 漏洞、前端没有 lint/test 门禁。

亮点：后端默认 DRF 鉴权、分页、异常包络、租户 manager 和跨租户测试做得比较扎实；抽样后端测试 `apps.devices.tests.test_device_authorization_api`、`apps.ai_models.tests.test_chat_api`、`apps.resources.tests.test_business_metadata_cache_api` 共 33 个用例通过。前端 TS strict 开启，统一 axios 客户端和 Vite build 也能通过。

优先修复：先拆掉默认超管凭据与过宽环境注入，强制生产 settings fail-fast；然后修设备激活唯一性和 npm audit high 项；再补齐前端 lint/test 脚本、OpenAPI serializer 标注和大组件拆分。

总评：C overall - 项目已经具备可运行的主干和不错的后端隔离测试，但安全发布门禁还没有收紧；按稳定公开发布标准，需要先完成 P0/P1 修复。

🔍 详细发现

1. compose 无条件创建已知超管账号 admin/admin123456

CriticalConfirmed
📁 Security / Release📍 docker-compose.yaml:57🎯 backend startup command
问题：启动命令在任何使用该 compose 的环境里都会创建同一组公开写在仓库里的 superuser。稳定公开发布时，这属于凭据写入源码/运行配置的高危面；只要部署者未立即轮换，公网后台就存在已知入口。
docker-compose.yaml:57 User.objects.filter(username='admin').exists() or User.objects.create_superuser('admin', 'admin@example.com', 'admin123456') CLAUDE.md:17 也记录了首次启动创建 superuser admin / admin123456 AGENTS.md:73 要求保留该 inline shell，说明它已经成为默认开发约定
MINIMAL FIX
把默认超管创建限制到显式开发开关，例如 CREATE_DEV_SUPERUSER=1；生产 settings 下如果仍启用则启动失败。把密码改为环境变量或一次性随机生成并只写到启动日志/密钥管理系统。
⏱ 估计：0.5-1 天  ·  回归测试：compose 注入 DJANGO_SETTINGS_MODULE=config.settings.prod 且未设置 CREATE_DEV_SUPERUSER 时，启动脚本不得创建 admin；开发开关打开时只在空库创建。

2. 当前运行时未加载 prod 安全设置，生产默认过宽

HighConfirmed
📁 Configuration / Security📍 backend/config/settings/base.py:31-217🎯 Django settings
问题：当前容器运行时 settings module 为 `config.settings.base`，因此 `prod.py` 的 secure cookies 没生效；同时 base 里 `SECRET_KEY` 有默认值、`ALLOWED_HOSTS` 默认 `*`、`CORS_ALLOW_ALL_ORIGINS` 默认 True 且允许 credentials。`check --deploy` 确认 HSTS、SSL redirect、secure cookies、SECRET_KEY 等发布检查未达标。
backend/config/settings/base.py:31 SECRET_KEY = os.getenv('DJANGO_SECRET_KEY', 'django-insecure-change-me') backend/config/settings/base.py:33 ALLOWED_HOSTS defaults to '*' backend/config/settings/base.py:207 CORS_ALLOW_ALL_ORIGINS defaults to True backend/config/settings/base.py:217 CORS_ALLOW_CREDENTIALS = True docker compose exec backend shell: SETTINGS_MODULE=config.settings.base; SESSION_COOKIE_SECURE=False; CSRF_COOKIE_SECURE=False docker compose exec backend python manage.py check --deploy: security.W004/W008/W009/W012/W016 and 38 total warnings
MINIMAL FIX
生产 compose / 部署环境必须设置 DJANGO_SETTINGS_MODULE=config.settings.prod；prod.py 中对 DJANGO_SECRET_KEY、DJANGO_ALLOWED_HOSTS、CORS_ALLOWED_ORIGINS 做 fail-fast 校验，并关闭 CORS_ALLOW_ALL_ORIGINS 默认值。
⏱ 估计：1 天  ·  回归测试：新增 settings 测试：prod 缺 SECRET_KEY/ALLOWED_HOSTS 时抛 ImproperlyConfigured；`docker compose exec backend python manage.py check --deploy --fail-level WARNING` 作为发布门禁。

3. db 容器接收完整 backend/.env，最小权限边界失效

HighConfirmed
📁 Security / Configuration📍 docker-compose.yaml:16-17🎯 db env_file
问题：`db` 服务通过 `env_file: ./backend/.env` 加载完整后端环境，实际只需要 POSTGRES_DB/USER/PASSWORD。`docker compose config` 会把所有 env_file 内容展开到 db/backend/celery 服务，包括 Django secret、云 API key、Webhook、MinIO secret 等类别；数据库容器和诊断输出都获得了超出职责的密钥面。
docker-compose.yaml:16-17 db: env_file: - ./backend/.env docker compose config 已确认 db.environment 中出现多类非数据库密钥；报告刻意不写入任何明文值。
MINIMAL FIX
给 db 单独使用 POSTGRES_DB/POSTGRES_USER/POSTGRES_PASSWORD 三个变量，或拆出 backend/.env 与 db.env。backend/celery 继续读取业务 env；避免让数据库容器持有云厂商和应用层密钥。
⏱ 估计：0.5 天  ·  回归测试：`docker compose config` 中 db.environment 只允许 POSTGRES_*；新增脚本检查 db 不含 API_KEY/SECRET/WEBHOOK/DJANGO_SECRET_KEY。

4. 设备首次激活可并发创建同设备码重复记录

HighConfirmed
📁 Stability / Backend API📍 backend/apps/devices/models.py:154, views.py:360-387🎯 DeviceActivationView.post
问题：设备唯一约束是 `(tenant, code)`，但首次激活创建的未绑定设备 `tenant=NULL`。PostgreSQL 唯一约束允许多个 NULL 组合；并发首次激活同一 deviceCode 时，两个事务都可能在 `select_for_update().filter(code=...)` 查不到行后各自创建记录，之后运行时会因重复设备码进入 409/不可用状态。
backend/apps/devices/models.py:154 UniqueConstraint(fields=['tenant', 'code'], name='unique_device_code_per_tenant') backend/apps/devices/views.py:360-361 select_for_update().filter(code=device_code) backend/apps/devices/views.py:387 Device.objects.create(code=device_code, tenant remains null until binding) backend/apps/devices/views.py:458 validate_device later rejects duplicates with len(devices) > 1
MINIMAL FIX
为未绑定设备增加条件唯一约束 UniqueConstraint(fields=['code'], condition=Q(tenant__isnull=True))，或按业务语义把 Device.code 设为全局唯一。激活接口改为数据库约束兜底 + IntegrityError 后重新读取。
⏱ 估计：1 天  ·  回归测试：使用 TransactionTestCase 并发提交相同 deviceCode，只允许一条 Device，第二个请求返回已有设备或可恢复响应。

6. Sentry 全环境启用 send_default_pii 且 DSN 硬编码

MediumConfirmed
📁 Security / Observability📍 backend/config/settings/base.py:12-16🎯 sentry_sdk.init
问题：Sentry 在 base settings import 时无条件初始化，DSN 写死在源码里，`send_default_pii=True` 会采集请求头/IP 等默认 PII。无环境开关、无采样策略、无 dev/prod 隔离时，本地或测试异常也可能进入同一个 Sentry 项目。
backend/config/settings/base.py:12 sentry_sdk.init(...) backend/config/settings/base.py:13 dsn="..." hardcoded DSN backend/config/settings/base.py:16 send_default_pii=True backend/config/sentry.py: before_send 只过滤少量 Celery 预期错误
MINIMAL FIX
改为 SENTRY_DSN 环境变量，缺省不初始化；send_default_pii 默认为 False，只在合规确认后按环境启用。保留 before_send，但增加敏感字段 scrubber。
⏱ 估计：0.5 天  ·  回归测试：无 SENTRY_DSN 时 sentry_sdk 不初始化；启用时 before_send 会过滤 Authorization、Cookie、API key、Webhook 字段。

7. 业务缓存注册表无过期且按完整 URL 累积

MediumConfirmed
📁 Performance / Stability📍 backend/config/business_cache.py:68-95🎯 CachedBusinessResponseMixin
问题：响应缓存本身有默认 300 秒 TTL，但注册表 `business-cache:<namespace>:registry` 使用 `timeout=None` 永久保存所有生成过的 cache key。cache key 原料包含 `request.get_full_path()`，搜索词、分页、host 和参数组合越多，注册表越大；过期的响应 key 不会自动从注册表剪枝。
backend/config/business_cache.py:68 raw_key includes request.get_full_path() backend/config/business_cache.py:84 cache.set(cache_key, data, timeout=timeout or get_business_cache_timeout()) backend/config/business_cache.py:95 cache.set(registry_key, sorted(registered_keys), timeout=None)
MINIMAL FIX
注册表改为有界集合或同 TTL；清理时忽略不存在 key。更稳妥做法是按 namespace 使用版本号失效：cache key 加 namespace version，写入时 bump version，避免维护永久 key 列表。
⏱ 估计：1 天  ·  回归测试：连续访问 200 个不同 keyword 后，registry 大小不超过设定上限或过期后归零；缓存失效仍能让列表刷新。

8. 聊天发送完整历史且 maxTokens 上限过大

MediumConfirmed
📁 Performance / Stability📍 backend/apps/ai_models/views.py:956-963, serializers.py:378-493🎯 ChatConversationViewSet.send
问题：聊天接口把当前会话所有历史消息一次性取出并发给上游 LLM，没有 token/window 上限、摘要裁剪或消息条数限制；同时 maxTokens 允许到 320000。长会话会线性放大数据库读取、请求体、上游费用、延迟和失败概率。
backend/apps/ai_models/views.py:956 history_messages = list(conversation.messages.order_by('created_at').values_list('role', 'content')) backend/apps/ai_models/views.py:963 api_messages.extend(...) backend/apps/ai_models/serializers.py:380/493 maxTokens 必须在 1 到 320000 之间
MINIMAL FIX
按模型配置建立最大上下文预算：保留 system prompt + 最近 N 轮 + summary，超过预算时拒绝或摘要压缩；maxTokens 上限按模型能力和租户策略配置。
⏱ 估计：1-2 天  ·  回归测试：创建 500 条历史消息后发送，断言上游 mock 收到的 messages 数/字符数在预算内，且摘要仍保留。

9. 上游异常文本直接回传到 SSE/ASR 客户端

MediumConfirmed
📁 Security / Stability📍 backend/apps/ai_models/views.py:1311-1313, realtime_asr.py:96🎯 LLM/ASR error handling
问题：通用异常会把 `str(exc)` 的前 100/200 字符写入聊天消息或通过 SSE/ASR WebSocket 返回给前端。上游 SDK/HTTP 异常可能包含 URL、代理、证书、请求片段或供应商实现细节，客户端不需要这些上下文。
backend/apps/ai_models/views.py:1311 full_content + f'[发生错误: {str(exc)[:100]}]' backend/apps/ai_models/views.py:1313 yield error content str(exc)[:200] backend/apps/ai_models/realtime_asr.py:96 json.dumps({'type': 'asr.error', 'message': str(exc)[:200]})
MINIMAL FIX
客户端统一返回稳定错误码和用户可读短消息；详细异常只进结构化日志/Sentry，并附 correlation id。保存到 ChatMessage 的失败内容也使用脱敏模板。
⏱ 估计：0.5 天  ·  回归测试：mock 上游抛出包含 URL/API key-like 字符串的异常，断言 SSE 和数据库消息不含异常原文，只含错误码。

12. OpenAPI 生成对大量 APIView 降级忽略
