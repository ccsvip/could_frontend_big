Evidence-based diagnosis

四类错误分别意味着什么

阻断恢复

1. 父表无法删除

cannot drop table resources_scrollingtext because other objects depend on it

原因：脚本第 438673 行先删除父表 resources_scrollingtext，到第 438700 行才删除子表 resources_scrollingtextitem。外键还在，且语句没有 CASCADE，所以父表删除必然失败。

后果：旧表保留，后面的 CREATE TABLE 和 INSERT 继续执行，目标库进入半旧半新状态。

目标库已被写入

2. 唯一键重复

Key (tenant_id, permissionpoint_id)=(1, 80) already exists

原因：备份文件中这个组合只出现一次，记录是 VALUES (787, 1, 80)。重复来自目标库：要么第一次失败后直接重跑，要么后台启动时执行 Django migrations，提前写入了默认公司的权限关系。

后果：说明恢复前提“空库且无写入者”没有成立。不要用 ON CONFLICT DO NOTHING 掩盖，它会留下来源不明的混合数据。

Navicat DDL 兼容问题

3. identity sequence 所有权

cannot change ownership of identity sequence

原因：表字段已经通过 GENERATED BY DEFAULT AS IDENTITY 绑定序列，Navicat 后面又执行 ALTER SEQUENCE ... OWNED BY。PostgreSQL 不允许再次改变 identity sequence 的所有权关系。

后果：单看这一条通常不是数据损坏的主因，但它证明该 SQL 的对象定义方式不适合做跨库恢复。

扩展对象导出错误

4. pgvector 内部类型与函数

CREATE OR REPLACE FUNCTION ... AS '$libdir/vector'

原因：Navicat 把 vector、halfvec、sparsevec 以及 C 函数当普通 schema 对象导出，但文件里没有 CREATE EXTENSION vector。这些对象应由扩展管理，不能靠手写对象顺序重建。

后果：未安装扩展时函数不存在；已安装扩展时又会与扩展拥有的类型冲突。两头都不可靠。

Plan A / Navicat first

方案 A：优先使用 Navicat 的“备份 / 还原”

适合你的使用习惯 保留 Navicat 图形化操作，但必须改用 PostgreSQL 的 Backup/Restore 工作流。不要再选“转储 SQL 文件 → 结构和数据”，也不要用“运行 SQL 文件”重放当前 149.5 MB 文件。
Navicat 功能本次是否使用原因
备份 / Backup使用使用 PostgreSQL 备份语义记录对象依赖和扩展关系。
还原备份 / Restore Backup使用与 Backup 配套，适合跨连接恢复。
转储 SQL 文件 / Dump SQL File不要使用当前失败文件正是此功能生成，扩展和 identity sequence 处理不可靠。
运行 SQL 文件 / Execute SQL File不要用于当前文件出错后会继续执行，留下半恢复数据库。
数据传输 / Data Transfer不作为首选迁移、扩展、identity 和自动种子数据同时存在，容易变成混合库。

  1. 建立目标 PostgreSQL 连接

    数据库参数来自目标项目配置，而不是 Linux 登录账号。当前克隆配置可确认：数据库名 digital_human、数据库用户 postgres、宿主机端口 5433。

    连接方式Navicat 常规页SSH 页
    直接连接 主机 192.168.182.129；端口 5433；初始数据库 postgres；用户 postgres；密码从目标 backend/.env 读取。 关闭 SSH 隧道。
    SSH 隧道 主机 127.0.0.1；端口 5433；初始数据库 postgres；数据库凭据同上。 主机 192.168.182.129；端口 22；SSH 用户 cancer；密码只在 Navicat 界面输入。
    若直接连接超时 先在目标服务器运行 docker compose ps，确认 solin_db 映射了 0.0.0.0:5433->5432/tcp；再检查虚拟机防火墙。不要为了图省事把数据库密码写进网页或聊天记录。

  2. 在源连接重新生成 Navicat 备份

    连接源主机 192.168.3.90 的 digital_human。在对象树选中数据库后，打开 备份 / Backup，选择 新建备份 / New Backup。常见版本会生成 Navicat 备份文件（例如 .nb3）。

    • 对象范围选择整个 digital_human 数据库或完整 public schema。
    • 保留表结构、数据、约束、索引、序列、视图和扩展信息。
    • 如果界面提供“所有者 / Owner”和“权限 / Privileges”，取消恢复源所有者与 ACL，目标统一使用当前数据库用户。
    • 备份结束后必须显示成功，并记录文件路径、大小和时间。
    看不到“备份 / Backup”或只能生成同类 SQL？ 立即停止方案 A，直接使用方案 B。不要手工从 149.5 MB SQL 中删除 vector 函数，这不仅费劲，而且很容易漏掉操作符、类型、聚合、访问方法和序列语句。

  3. 在目标连接创建真正的空库

    目标 Navicat 连接必须先连维护数据库 postgres，不能连着 digital_human 删除自己。下面三段 SQL 逐条执行，不要包在事务中。

    Navicat 查询窗口 / 连接到 postgres / 第 1 条
    DROP DATABASE IF EXISTS digital_human WITH (FORCE);
    Navicat 查询窗口 / 第 2 条
    CREATE DATABASE digital_human
  WITH OWNER = postgres
       ENCODING = 'UTF8'
       TEMPLATE = template0;

    刷新数据库列表，再打开新建的 digital_human 查询窗口，执行：

    Navicat 查询窗口 / 连接到 digital_human / 第 3 条
    CREATE EXTENSION IF NOT EXISTS vector;

SELECT extname, extversion
FROM pg_extension
WHERE extname = 'vector';

SELECT count(*) AS public_table_count
FROM pg_tables
WHERE schemaname = 'public';

    此时应看到一条 vector 扩展记录，且 public_table_count = 0。扩展创建类型和函数，但不会创建项目业务表。

  4. 把新备份还原到目标库

    选中目标 digital_human，打开 备份 / Backup → 还原备份 / Restore Backup，选择第 2 步生成的备份文件。不同 Navicat 版本菜单可能显示为 Restore 或 Restore Backup。

    • 目标数据库必须明确显示为 192.168.182.129 / digital_human。
    • 如果有 Exit on error / 出错即停止，必须开启。
    • 如果有 No owner / 不恢复所有者、No privileges / 不恢复权限，必须开启。
    • 空库恢复时不需要“只恢复数据”或“仅追加数据”。
    • 恢复窗口出现第一条 ERROR 就停止，不允许忽略错误后继续完成。
    扩展已存在通常不是问题 标准备份会以扩展为单位记录 pgvector，不会重建 $libdir/vector 内部函数。如果 Navicat 备份仍尝试逐个创建这些内部函数，说明该版本的 Backup 也不适配，改走方案 B。

  5. 先验证数据库，再启动项目

    不要看到 Navicat 进度条 100% 就直接启动。先执行“恢复验证”章节的 SQL，所有硬性项目通过后，才在目标虚拟机启动服务：

    目标虚拟机 / Linux Shell

docker compose up -d backend celery_worker celery_beat

docker compose logs --no-color --tail 120 backend

    后台日志中允许出现“没有待应用迁移”；不允许出现数据库连接失败、迁移表缺失、vector 类型不存在或唯一键冲突。

Plan B / Recommended

方案 B：用 PostgreSQL 原生工具生成标准备份

这是技术上最稳妥的方案 使用源数据库容器内的 PostgreSQL 16 pg_dump 创建 custom-format 归档，再用目标容器内的 pg_restore 恢复。对象依赖、扩展、序列和数据顺序由 PostgreSQL 自己处理。

  1. 在源项目容器内生成 custom-format 备份

    在 Windows 源项目目录打开 PowerShell。备份先写到数据库容器内部，再用 docker cp 取出，避免 Windows PowerShell 5 把二进制重定向搞坏。

    源主机 192.168.3.90 / PowerShell
    Set-Location "C:/code/SVN_CODE/branches/real/could_frontend"

docker compose exec -T db sh -lc 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" -Fc --no-owner --no-privileges -f /tmp/digital_human.dump'

docker compose exec -T db sh -lc 'pg_restore -l /tmp/digital_human.dump | head -n 30'

docker cp "solin_db:/tmp/digital_human.dump" "C:/云端静态资源备份+.env+数据库备份/数据库备份/192.168.3.90/digital_human.dump"

Get-Item "C:/云端静态资源备份+.env+数据库备份/数据库备份/192.168.3.90/digital_human.dump"
Get-FileHash -Algorithm SHA256 "C:/云端静态资源备份+.env+数据库备份/数据库备份/192.168.3.90/digital_human.dump"

    pg_restore -l 应输出对象清单，文件大小不能为 0。命令从容器环境读取数据库用户和库名，不会把数据库密码写进命令历史。

  2. 把备份传到目标虚拟机

    使用 Windows 自带 OpenSSH 的 scp。命令执行后会交互提示 SSH 密码，直接输入即可；不要把密码拼在命令里。

    源主机 / PowerShell
    scp "C:/云端静态资源备份+.env+数据库备份/数据库备份/192.168.3.90/digital_human.dump" "cancer@192.168.182.129:/home/cancer/workspace/could_frontend/digital_human.dump"

    若 PowerShell 提示找不到 scp，安装 Windows“OpenSSH 客户端”可选功能，或者用 WinSCP 把同一个文件传到相同目标路径。

  3. 在目标端确认文件和运行环境

    目标虚拟机 192.168.182.129 / Linux Shell
    cd "/home/cancer/workspace/could_frontend"

pwd
ls -lh "./digital_human.dump"
sha256sum "./digital_human.dump"
docker compose ps
grep -E '^(DB_PORT|POSTGRES_DB|POSTGRES_USER)=' ".env" "backend/.env"

    期望看到路径完全匹配、dump 文件大小合理、数据库服务 healthy，配置中数据库名为 digital_human、用户为 postgres。不要输出或截图 POSTGRES_PASSWORD。

  4. 停止写服务并备份当前目标库

    即使目标库“不要了”，也建议先做最后一份回退备份。该文件只保存在目标数据库容器和项目目录。

    目标虚拟机 / Linux Shell
    docker compose stop backend celery_worker celery_beat

docker compose exec -T db sh -lc 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" -Fc --no-owner --no-privileges -f /tmp/target-before-restore.dump'

docker cp "solin_db:/tmp/target-before-restore.dump" "./target-before-restore.dump"

ls -lh "./target-before-restore.dump"
docker compose ps
    下一步不可逆地替换目标库 请再次核对终端提示符、pwd、服务器 IP、POSTGRES_DB 和回退文件。确认无误后再执行。

  5. 删除旧库并创建空库

    dropdb --force 会终止目标库连接；写服务已经停止，所以不会与迁移或业务请求抢占。

    目标虚拟机 / Linux Shell / 危险操作
    docker compose exec -T db sh -lc 'dropdb --force --if-exists -U "$POSTGRES_USER" "$POSTGRES_DB"'

docker compose exec -T db sh -lc 'createdb -U "$POSTGRES_USER" -O "$POSTGRES_USER" -E UTF8 "$POSTGRES_DB"'

docker compose exec -T db sh -lc 'psql -X -v ON_ERROR_STOP=1 -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "SELECT current_database(), current_user;"'

  6. 把归档送入容器并恢复

    --exit-on-error 是关键：任何一条对象或数据错误都会立即停止，不再制造“表面完成、内部半残”的数据库。

    目标虚拟机 / Linux Shell
    docker cp "./digital_human.dump" "solin_db:/tmp/digital_human.dump"

docker compose exec -T db sh -lc 'pg_restore -U "$POSTGRES_USER" -d "$POSTGRES_DB" --clean --if-exists --no-owner --no-privileges --exit-on-error /tmp/digital_human.dump'

docker compose exec -T db sh -lc 'psql -X -v ON_ERROR_STOP=1 -U "$POSTGRES_USER" -d "$POSTGRES_DB" -c "SELECT count(*) AS tables FROM pg_tables WHERE schemaname = '\''public'\'';"'

    pg_restore 退出码必须是 0。只要出现 ERROR，就保持服务停止，回到“删除旧库并创建空库”重新开始；不要在当前库上继续补执行。

  7. 完成验证并启动服务

    先执行下一章节的数据库验证 SQL；确认全部通过后再启动：

    目标虚拟机 / Linux Shell
    docker compose up -d backend celery_worker celery_beat

docker compose ps
docker compose logs --no-color --tail 150 backend

docker compose exec backend python manage.py migrate --check

    migrate --check 应返回 0 且不创建新迁移。恢复后的 django_migrations 已记录源库迁移状态，后台启动不应重新灌入默认权限。

必须保留 .sql 文件时怎么办？

可以让 PostgreSQL 自己生成 plain-format SQL，而不是使用当前 Navicat SQL。源容器内改用：

源数据库容器 / 生成标准 plain SQL
docker compose exec -T db sh -lc 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" -Fp --clean --if-exists --no-owner --no-privileges --encoding=UTF8 -f /tmp/digital_human.pg_dump.sql'

docker cp "solin_db:/tmp/digital_human.pg_dump.sql" "./digital_human.pg_dump.sql"

目标端恢复必须用 psql -X -v ON_ERROR_STOP=1，不能让错误被忽略。对于 149 MB 且包含大量向量的数据，custom-format 通常更快、更可控，因此仍推荐 .dump。

Verification matrix

恢复成功不是“没有红字”，而是这些检查全部通过

以下 SQL 可以在 Navicat 的目标 digital_human 查询窗口执行；也可以通过容器内 psql 执行。

目标数据库 / 验收 SQL
-- 1. 当前连接必须是目标库
SELECT current_database(), current_user, version();

-- 2. pgvector 必须由扩展管理
SELECT extname, extversion
FROM pg_extension
WHERE extname = 'vector';

-- 3. 项目表与迁移记录必须存在
SELECT count(*) AS public_table_count
FROM pg_tables
WHERE schemaname = 'public';

SELECT count(*) AS migration_count, max(applied) AS latest_migration_time
FROM django_migrations;

-- 4. 关键业务数据数量，用源库同一查询结果做对比
SELECT count(*) AS tenant_count FROM tenants_tenant;
SELECT count(*) AS permission_count FROM accounts_permissionpoint;
SELECT count(*) AS tenant_permission_links FROM tenants_tenant_permission_points;
SELECT count(*) AS resource_count FROM resources_resource;
SELECT count(*) AS knowledge_chunk_count FROM knowledge_base_knowledgedocumentchunk;

-- 5. 该查询必须返回 0 行
SELECT tenant_id, permissionpoint_id, count(*) AS duplicate_count
FROM tenants_tenant_permission_points
GROUP BY tenant_id, permissionpoint_id
HAVING count(*) > 1;

-- 6. 向量数据应可正常读取
SELECT
  count(*) FILTER (WHERE embedding_vector IS NOT NULL) AS vectorized_chunks,
  max(vector_dims(embedding_vector)) AS max_vector_dimensions
FROM knowledge_base_knowledgedocumentchunk;
检查项通过标准失败时处理
current_database()digital_human立即停止，连接错库。
vector 扩展正好 1 行，显示版本号确认目标镜像是 pgvector/pgvector:pg16，再重建空库恢复。
public 表数量大于 0，且与源库同类查询一致备份或恢复不完整，不能启动后台。
django_migrations有记录，最大时间非空恢复的不是完整项目数据库。
关键表数量逐项与源库查询结果一致不要补插；重新生成备份并恢复。
重复权限关系返回 0 行目标库不是空库或发生并发写入，重做。
vector_dims()有向量时返回正整数检查 pgvector 版本、列类型和恢复错误。
migrate --check退出码 0确认代码版本与源库迁移版本完全一致。
容器健康db/backend/worker/beat 正常运行看对应容器最近 150 行日志。
必须做源端数量对比 “表里有数据”不等于“数据完整”。在源库和目标库分别执行同一组 count(*)，至少核对公司、权限、素材、知识切片、设备、会话等核心表。

Troubleshooting

看到这些错误时怎么处理

错误或现象真正含义正确动作
cannot drop table ... depends on it 仍在使用错误的 Navicat SQL，或目标库不是空库。 停止恢复；不要加 CASCADE 修文件；重新建空库并用 Backup/Restore 或 pg_restore。
duplicate key value violates unique constraint 目标库已有数据，常见原因是后台迁移、种子命令或失败后重跑。 确认三个写服务停止；删除并重建目标库；从头恢复。
cannot change ownership of identity sequence Navicat 对 identity sequence 生成了冗余 DDL。 放弃当前 SQL；不要逐条忽略。用 PostgreSQL 标准备份。
type "vector" does not exist 目标镜像缺少 pgvector，或扩展未被备份/恢复。 确认 db 镜像是 pgvector/pgvector:pg16；在空库确认可创建扩展后重做。
could not access file "$libdir/vector" 目标 PostgreSQL 容器没有 vector 共享库，通常用了官方 postgres 镜像。 恢复项目的 pgvector 镜像配置并重建 db 容器；不要手工复制 .so。
unsupported version in file header 目标 pg_restore 比生成 dump 的 pg_dump 老。 确保源、目标均使用 PostgreSQL 16 容器内工具。
permission denied to create extension 恢复用户不是具备扩展权限的数据库管理员。 用目标容器的 POSTGRES_USER 恢复；不要给普通业务账号临时超级权限。
Navicat 显示 Finished with error 恢复没有成功，即使后面部分语句执行完了。 把当前库当作不可用；保存日志；重建空库后重做。
数据库正常但图片/视频 404 SQL 只恢复对象键，没有迁移 MinIO 桶或本地 media 文件。 迁移对应存储，并核对目标 .env 的 endpoint、bucket 与凭据。

Beyond PostgreSQL

数据库恢复完，不代表项目数据已经全部迁完

1. MinIO / 素材文件

项目启用了 MinIO，数据库中的 resources/2026/... 等值只是对象键或路径。PostgreSQL 备份不会包含图片、视频、模型包和其他二进制对象。

  • 单独备份并迁移源 MinIO 的 digital-human 桶。
  • 迁移前后比较对象数量和总容量，并随机打开图片、视频验证。
  • 目标 backend/.env 中的 MinIO endpoint 必须从容器内可达。
  • 访问密钥和 Secret 只放在目标环境配置或密钥管理中，不写进 HTML、命令脚本或 Git。

2. 本地 media 目录

如果有未进入 MinIO 的历史文件，还要检查源项目 backend/media/。保持目录结构复制到目标对应路径，并保留文件所有者和读取权限。

3. .env 配置

可以拿源端 .env 和 backend/.env 做对照，但不要整文件无脑覆盖目标：

  • 数据库名与用户可以一致；主机、端口和容器内 endpoint 要按目标环境配置。
  • 第三方 API Key、MinIO 密钥、JWT Secret 等敏感项要通过安全通道迁移。
  • 目标虚拟机特有的域名、TLS 路径、回调 URL、跨域来源不能照搬源主机。
  • 迁移后不要把包含真实密钥的 .env 提交到 Git。
数据一致性提醒 如果源系统仍在写入，数据库快照与 MinIO 文件复制可能不是同一时刻。正式切换时应安排短暂停写窗口：停止源端上传与业务写入，完成最终数据库 dump 和 MinIO 增量同步，再启动目标端。
