一、现状分析

当前知识库系统在多模态内容检索上存在严重瓶颈，核心函数 _score_media_asset() 采用纯关键词匹配，无法理解图片/视频的语义内容，导致多模态检索准确率远低于文本检索。

1
图片无语义向量
当前图片仅通过文件名、alt文本等元数据生成文本向量，图片本身视觉内容（场景、物体、颜色、布局）完全未被编码。一张包含"会议白板"的图片，如果文件名是 IMG_2024.png，检索"会议记录"将完全无法命中。
影响范围
所有含图片的知识库文档，图片检索召回率 < 30%
2
向量存储性能差
KnowledgeMediaAsset.embedding 使用 JSONField 存储向量，每次相似度计算需要全表扫描加载所有向量到 Python 内存，通过 _cosine_similarity() 手动计算。当素材量达到 1 万+时，单次检索耗时从毫秒级退化到秒级。
影响范围
素材量 > 5000 时，检索延迟 > 2s；无索引支持，无法利用数据库层优化
3
多模态对齐缺失
文本向量与图片向量在不同的向量空间中生成（文本用 text-embedding-v3，图片用 CLIP），查询文本无法直接与图片向量进行相似度比较，需要跨模态对齐机制将两者映射到同一语义空间。
影响范围
"文搜图"场景完全不可用，用户必须用精确文件名才能找到图片
4
视频理解完全缺失
视频文件当前仅作为附件存在，没有任何帧提取、内容理解或时序索引。30 分钟的产品演示视频，用户无法搜索其中某一帧或某一段的内容。
影响范围
视频素材检索召回率 = 0%，全量视频内容无法被RAG利用
5
素材关联粗糙
_score_media_asset() 仅通过文件名关键词匹配关联，无语义理解。同一张图片被多个文档引用时，无法区分上下文相关性；不同文档中语义相似但文件名不同的图片无法建立关联。
影响范围
素材推荐精确度低，跨文档素材发现能力为零
问题影响总结
核心瓶颈
_score_media_asset() 的纯关键词匹配模式是所有问题的根源 — 它无法理解"图片里有什么"，导致下游所有检索、推荐、关联功能失效。

二、方案对比

针对多模态检索问题，我们评估了三种技术路线。每种方案在检索质量、实施成本、维护复杂度上各有权衡。

方案A：多模态Embedding直接向量化
中等成本

使用多模态Embedding模型（如 Chinese-CLIP、DashScope multimodal-embedding）直接将图片和文本编码到同一向量空间，实现跨模态语义检索。

Python · 图片向量化示例Chinese-CLIP
import torch from transformers import ChineseCLIPModel, ChineseCLIPProcessor # 加载多模态Embedding模型 model = ChineseCLIPModel.from_pretrained("OFA-Sys/chinese-clip-vit-base-patch16") processor = ChineseCLIPProcessor.from_pretrained("OFA-Sys/chinese-clip-vit-base-patch16") # 图片 → 向量 from PIL import Image image = Image.open("meeting_whiteboard.jpg") inputs = processor.images(image, return_tensors="pt") img_emb = model.get_image_features(**inputs) img_emb = img_emb / img_emb.norm(dim=-1, keepdim=True) # 文本 → 向量（同一空间） text_inputs = processor.text(["会议白板记录"], return_tensors="pt") txt_emb = model.get_text_features(**text_inputs) txt_emb = txt_emb / txt_emb.norm(dim=-1, keepdim=True) # 跨模态相似度 similarity = (txt_emb @ img_emb.T).item() # ≈ 0.85

优势

  • 直接跨模态检索，"文搜图"开箱即用
  • 向量空间统一，检索延迟低
  • 社区活跃，Chinese-CLIP 开源可用

劣势

  • 向量维度固定，模型升级需全量重算
  • 对复杂场景描述能力有限（如"第三排左侧的人"）
  • 视频需额外拆帧处理
  • 自部署模型需GPU资源
方案B：VLM描述生成 + 文本向量化
推荐首选

使用视觉语言模型（VLM）为图片/视频帧生成自然语言描述，再通过现有文本Embedding管道向量化，与文档Chunk在同一向量空间检索。

Python · DashScope VLM 描述生成DashScope API
import dashscope from dashscope import MultiModalConversation # VLM 图片描述生成 response = MultiModalConversation.call( model="qwen-vl-max", messages=[{ "role": "user", "content": [ {"image": "https://example.com/meeting.jpg"}, {"text": "请详细描述这张图片的内容，包括场景、物体、文字信息。"} ] }] ) description = response["output"]["choices"][0]["message"]["content"] # → "一间会议室中，白板上写满了项目排期和架构图， # 桌上散放着笔记本电脑和咖啡杯，有5人正在讨论..." # 复用现有文本向量化管道 embedding = get_text_embedding(description) # 走现有 text-embedding-v3

优势

  • 复用现有文本Embedding管道，改造成本最低
  • VLM描述语义丰富，理解"白板内容"等抽象概念
  • API调用无需GPU，DashScope按量计费
  • 描述可人工审核/编辑，可控性强

劣势

  • 描述生成有信息损失（视觉细节→文本）
  • VLM推理延迟较高（单图2-5秒）
  • API调用有成本（约 ¥0.02/图）
  • 大批量处理需异步队列
方案C：VLM描述 + 多模态Embedding混合
最优质量

结合方案A和B的优势：VLM描述用于文本检索精排，多模态Embedding用于跨模态粗排，两层检索结果融合排序。

三层混合检索架构
graph TB Q[用户查询] --> TA[文本Embedding] Q --> MA[多模态Embedding] Q --> KW[关键词匹配] TA --> TR[文本向量检索] MA --> MR[多模态向量粗排] KW --> KR[关键词兜底] MR --> F[结果融合排序] TR --> F KR --> F F --> R[最终结果] style F fill:#8b5cf6,color:#fff style R fill:#10b981,color:#fff

优势

  • 检索质量最高，三层互补
  • 粗排+精排兼顾速度与精度
  • 关键词兜底保证最低召回率
  • 可持续扩展新检索通道

劣势

  • 实施复杂度最高，需维护三套管道
  • 多模态向量需GPU或API成本
  • 结果融合策略需调优
  • 存储成本增加（多套向量）

三、推荐方案设计

基于方案对比分析，推荐采用分阶段演进策略：先以最低成本验证VLM描述的价值，再逐步叠加多模态向量和视频理解能力。

def hybrid_multimodal_search(query_text, top_k=10): """双层混合检索：多模态粗排 + VLM精排""" # Layer 1: 多模态向量粗排（速度快） mm_results = pgvector_search( query=query_text, field="multimodal_embedding", top_k=top_k * 3 # 过采样 ) # Layer 2: VLM描述文本精排（精度高） vlm_results = pgvector_search( query=query_text, field="vlm_description_embedding", top_k=top_k * 3 ) # Layer 3: 关键词兜底 kw_results = keyword_search(query_text) # 融合排序：RRF (Reciprocal Rank Fusion) return reciprocal_rank_fusion( [mm_results, vlm_results, kw_results], weights=[0.4, 0.5, 0.1] )

5.2 新 _score_media_asset 算法设计

核心改造点：将纯关键词匹配替换为VLM描述向量检索 + 关键词兜底的混合策略。

Python · 新 _score_media_asset 算法retriever.py
def _score_media_asset(self, query: str, knowledge_base_id: int, top_k: int = 5): """混合多模态素材检索：VLM向量优先 + 关键词兜底""" results = [] # ========== Layer 1: VLM描述向量检索（主力） ========== try: query_embedding = get_text_embedding(query) vlm_hits = KnowledgeMediaAsset.objects.filter( knowledge_base_id=knowledge_base_id, vlm_description_embedding__isnull=False ).annotate( similarity=CosineDistance("vlm_description_embedding", query_embedding) ).filter( similarity__lt=0.3 # 相似度阈值 ).order_by("similarity")[:top_k] for hit in vlm_hits: results.append({ "asset": hit, "score": 1.0 - hit.similarity, "source": "vlm_embedding" }) except Exception as e: logger.warning(f"VLM向量检索失败: {e}") # ========== Layer 2: 关键词兜底 ========== if len(results) < top_k: keywords = extract_keywords(query) kw_hits = KnowledgeMediaAsset.objects.filter( knowledge_base_id=knowledge_base_id ).filter( Q(file_name__icontains=keywords[0]) | Q(alt_text__icontains=keywords[0]) ).exclude( pk__in=[r["asset"].pk for r in results] )[:top_k - len(results)] for hit in kw_hits: results.append({ "asset": hit, "score": 0.3, # 关键词匹配给较低分数 "source": "keyword_fallback" }) # 按分数排序返回 results.sort(key=lambda x: x["score"], reverse=True) return results[:top_k]

改造注意事项
  • VLM异步任务需设置超时（30s）和重试（3次），避免阻塞队列
  • pgvector索引建议使用 IVFFlat（数据量 < 100万）或 HNSW（数据量 > 100万）
  • 存量图片批量处理建议分批（每批100张），避免API限流
  • VLM描述字段需加入后台管理界面，支持人工审核和编辑
  • Phase3多模态向量与VLM向量使用不同字段，便于A/B对比
