# PRD：基于唯一设备码的语音问答 HTML 页面

## 1. 项目背景

当前系统已经具备设备管理、ASR、LLM、TTS、知识库等能力。为了让设备端、演示端或测试人员可以快速验证完整语音交互链路，需要提供一个轻量级前端 HTML 页面。

该页面不承担后台管理职责，只负责通过一个唯一设备码识别当前设备身份，然后调用对应接口完成语音问答流程。

用户打开页面后，可以输入或携带设备码，点击录音，说一段话，系统将语音上传到后端，由后端完成语音识别、问题处理、文本回答生成和语音合成，最后前端展示文本回复并播放语音回复。

---

## 2. 产品目标

### 2.1 核心目标

实现一个轻量级 HTML 页面，用于验证和演示设备侧语音问答能力。

核心流程如下：

```text
用户打开页面
↓
页面携带唯一设备码
↓
请求后端校验设备与获取配置
↓
用户点击录音并说话
↓
前端上传音频
↓
后端完成 ASR 识别
↓
后端调用 LLM / 知识库生成答案
↓
后端调用 TTS 生成语音
↓
前端展示文本答案并播放语音答案
```

### 2.2 业务目标

1. 降低设备语音问答链路的测试成本。
2. 提供一个不依赖后台管理系统的独立演示页面。
3. 验证设备码、ASR、LLM、TTS、知识库、资源配置是否能完整打通。
4. 为后续数字人设备端、小程序端、H5 端提供接口交互参考。

---

## 3. 用户角色

### 3.1 测试人员

使用该页面测试指定设备码是否可用，验证语音问答链路是否正常。

### 3.2 运维人员

通过页面快速判断设备配置、模型配置、语音识别、语音合成是否存在问题。

### 3.3 演示人员

通过页面向客户或内部人员展示“语音提问 → 智能回答 → 语音播报”的效果。

### 3.4 设备端开发人员

参考该 HTML 页面和文档，接入真实设备端或其他客户端。

---

## 4. 使用场景

### 场景一：通过 URL 携带设备码打开页面

用户访问：

```text
https://example.com/device-chat.html?deviceCode=DEVICE_001
```

页面自动读取 `deviceCode`，调用设备校验接口，获取设备信息和可用配置。

### 场景二：手动输入设备码

如果 URL 没有携带设备码，页面展示设备码输入框。

用户输入设备码后，点击“连接设备”，页面调用设备校验接口。

### 场景三：用户语音提问

用户点击“开始说话”，浏览器开始录音。

用户说完后点击“结束说话”，页面将录音文件上传给后端。

后端返回：

```text
识别出的用户问题
文本答案
语音答案地址或音频数据
```

前端展示文本内容，并自动播放语音。

## 6. 页面功能需求

## 6.1 页面初始化

页面加载后，需要执行以下逻辑：

1. 从 URL Query 中读取 `deviceCode`。
2. 如果存在 `deviceCode`，自动请求设备校验接口。
3. 如果不存在 `deviceCode`，展示设备码输入区域。
4. 设备校验成功后，展示语音问答主界面。
5. 设备校验失败后，展示错误提示。

### 页面状态

```text
未连接设备
连接中
设备连接成功
设备连接失败
录音中
处理中
回答完成
发生错误
```

---

## 6.2 设备码输入与校验

### 功能说明

设备码是当前页面识别设备身份的唯一凭证。

页面需要支持两种设备码来源：

```text
1. URL 参数：?deviceCode=xxx
2. 用户手动输入
```

### 交互要求

当页面未获取到设备码时，展示：

```text
请输入设备码
[ 输入框 ]
[ 连接设备按钮 ]
```

当设备码校验成功后，展示：

```text
当前设备：xxx
设备状态：已连接
```

当设备码校验失败后，展示：

```text
设备码无效或设备未授权，请检查设备码
```

---

## 6.3 麦克风授权

### 功能说明

用户点击“开始说话”时，页面需要请求浏览器麦克风权限。

### 交互要求

如果用户授权成功，开始录音。

如果用户拒绝授权，提示：

```text
无法访问麦克风，请在浏览器中允许麦克风权限后重试。
```

如果当前浏览器不支持录音能力，提示：

```text
当前浏览器不支持录音功能，请更换 Chrome、Edge 或 Safari 浏览器。
```

---

## 6.4 录音功能

### 功能说明

用户点击按钮后开始录音，再次点击后结束录音并上传。

### 交互设计

默认按钮：

```text
按下开始说话
```

录音中按钮：

```text
正在录音，点击结束
```

结束录音后：

```text
正在识别和生成回答...
```

### 录音格式

建议优先使用浏览器支持度较高的格式：

```text
audio/webm
audio/wav
audio/mp3
```

具体格式以后端 ASR 接口支持为准。

---

## 6.5 语音上传与处理

### 功能说明

录音结束后，前端将音频文件上传至后端接口。

后端处理流程：

```text
音频文件
↓
ASR 语音识别
↓
得到用户文本问题
↓
LLM / 知识库生成回答
↓
TTS 合成语音
↓
返回文本答案和语音答案
```

### 前端请求要求

请求时必须携带设备码。

推荐方式：

```text
Header: X-Device-Code: DEVICE_001
```

或者：

```text
FormData: deviceCode=DEVICE_001
```

为了安全和统一，建议以后端约定为准。

---

## 6.6 文本展示

接口返回后，页面需要展示两段文本：

```text
我说的问题：
今天有哪些注意事项？

系统回答：
今天需要注意设备电量、网络连接状态，以及保持麦克风正常工作。
```

### 展示要求

1. 用户原始问题展示在上方。
2. 系统答案展示在下方。
3. 如果 ASR 没有识别出内容，提示用户重新说一遍。
4. 如果回答生成失败，展示明确错误原因。

---

## 6.7 语音播放

### 功能说明

后端返回语音结果后，前端自动播放。

支持两种返回方式：

```text
1. audioUrl：语音文件地址
2. audioBase64：Base64 音频数据
```

优先建议使用 `audioUrl`，便于浏览器播放和缓存。

### 交互要求

1. 默认自动播放语音答案。
2. 如果浏览器阻止自动播放，展示“点击播放语音”按钮。
3. 提供重新播放按钮。
4. 语音加载失败时展示错误提示。

---

## 6.8 重新提问

回答完成后，用户可以继续点击按钮再次提问。

页面需要保留当前设备码，不需要用户重复输入。

## 8. 接口需求

以下为建议接口，最终路径以后端实际实现为准。

---

## 8.1 设备校验接口

### 接口用途

根据设备码校验设备是否存在、是否启用、是否有权限使用语音问答能力。

### 请求方式

```http
GET /api/v1/device/session
```

### 请求参数

```text
deviceCode: string
```

### 示例请求

```http
GET /api/v1/device/session?deviceCode=DEVICE_001
```

### 成功响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "deviceCode": "DEVICE_001",
    "deviceName": "大厅数字人设备",
    "tenantId": 1,
    "status": "active",
    "asrEnabled": true,
    "ttsEnabled": true,
    "llmEnabled": true
  }
}
```

### 失败响应

```json
{
  "code": 4001,
  "message": "设备码无效或设备未授权",
  "data": null
}
```

---

## 8.2 语音问答接口

### 接口用途

上传用户语音，由后端完成 ASR、LLM、TTS 处理，并返回文本与语音结果。

### 请求方式

```http
POST /api/v1/device/voice-chat
```

### 请求头

```http
X-Device-Code: DEVICE_001
```

### 请求格式

```text
multipart/form-data
```

### 请求参数

| 参数         | 类型     | 必填 | 说明               |
| ---------- | ------ | -: | ---------------- |
| audio      | File   |  是 | 用户录音文件           |
| sessionId  | string |  否 | 会话 ID，用于后续多轮对话   |
| format     | string |  否 | 音频格式，例如 webm、wav |
| sampleRate | number |  否 | 采样率              |

### 成功响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "sessionId": "session_123456",
    "questionText": "今天有哪些注意事项？",
    "answerText": "今天需要注意设备电量、网络连接状态，以及保持麦克风正常工作。",
    "audioUrl": "https://example.com/media/tts/session_123456.mp3",
    "audioBase64": null,
    "duration": 3.5,
    "traceId": "trace_abc123"
  }
}
```

### 失败响应

```json
{
  "code": 5001,
  "message": "语音识别失败，请重新说一遍",
  "data": {
    "traceId": "trace_abc123"
  }
}
```

## 12. 安全要求

1. 前端不得暴露任何 ASR、TTS、LLM 的 API Key。
2. 前端只允许携带设备码请求业务接口。
3. 设备码需要由后端校验权限。
4. 后端需要判断设备是否启用。
5. 后端需要判断设备所属租户是否有可用模型配置。
6. 接口需要限制音频大小。
7. 接口需要限制请求频率，防止设备码被滥用。
8. 建议返回 `traceId`，便于排查问题。
9. 建议接口支持 HTTPS，避免音频数据明文传输。

---

## 13. 性能要求

1. 单次录音建议限制在 60 秒以内。
2. 音频文件大小建议限制在 10MB 以内。
3. 页面点击结束录音后，应在 1 秒内展示“处理中”状态。
4. 正常情况下，完整回答返回时间建议控制在 3 到 10 秒内。
5. 如果超过 30 秒未返回，需要提示用户请求超时。
6. 音频播放需要尽量使用流畅的 `audioUrl` 方式。

## 16. 验收标准

### 16.1 基础功能验收

1. 页面可以正常打开。
2. URL 携带设备码时，页面可以自动识别设备码。
3. 未携带设备码时，页面可以手动输入设备码。
4. 设备码有效时，可以进入语音问答页面。
5. 设备码无效时，可以展示明确错误提示。
6. 用户可以点击按钮开始录音。
7. 用户可以点击按钮结束录音。
8. 录音结束后可以成功上传音频。
9. 页面可以展示 ASR 识别出的文本问题。
10. 页面可以展示系统生成的文本答案。
11. 页面可以播放系统生成的语音答案。
12. 用户可以连续进行多次提问。

### 16.2 异常场景验收

1. 用户拒绝麦克风权限时，页面有提示。
2. 网络异常时，页面有提示。
3. 后端返回 ASR 失败时，页面有提示。
4. 后端返回 LLM 失败时，页面有提示。
5. 后端返回 TTS 失败时，页面仍展示文本答案。
6. 接口超时时，页面有重试入口。
7. 设备码失效时，页面提示重新连接设备。

本项目需要实现一个轻量级 HTML 语音问答页面。页面通过唯一设备码识别设备身份，调用后端接口完成语音识别、智能问答和语音合成，最终向用户展示文本答案并播放语音答案。同时需要提供完整接入文档，方便测试人员、演示人员和设备端开发人员快速使用。
