# 第三方机器人上下文集成指南

## 目的

本文用于指导后续接入不规则第三方 LLM / chatbot 时，正确配置多轮上下文，避免出现“每轮都创建新会话”的问题。

核心原则只有一条：发给第三方的上游会话 ID 必须来自第三方 API 响应，不能由本系统生成。

本系统可以维护本地运行时会话，用于找到缓存的第三方变量；但本地会话 ID 不等于第三方会话 ID，不能直接发送给第三方。

## 两类会话 ID

### 本地运行时会话 ID

来源：安卓、WebSocket、HTTP 语音接口或后台运行时生成/传入。

用途：在本系统内找到同一轮设备或语音会话对应的 `ChatConversation.external_session`。

示例字段：

- HTTP 语音接口响应里的 `sessionId`
- WebSocket agent/realtime payload 里的 `sessionId`
- 后台页面调试里的 `conversationId`

禁止用途：不能把这些本地 ID 当成第三方 `chat_id`、`sessionId` 或其他上游上下文 ID 发送给第三方。

### 第三方上游会话 ID

来源：第三方 API 响应。

用途：后续请求第三方时保持上下文。

示例字段：

- 方案 A：华鹏接口返回 `data`，保存为 `chat_id`
- 方案 B：FlowMesh 接口返回 `data.sessionId`，保存为 `sessionId`
- 其他厂商：可能叫 `conversation_id`、`threadId`、`chat.session.id` 等，必须按配置提取

保存位置：`ChatConversation.external_session`。

## 配置模型

第三方机器人集成配置由多个 step 组成。每个 step 描述一次第三方 HTTP 调用。

常用字段：

| 字段 | 说明 |
| --- | --- |
| `key` | 步骤标识，建议可读，如 `open_chat`、`send_message` |
| `method` | HTTP 方法 |
| `path` | 第三方路径，支持 `{{变量}}` 模板 |
| `headers` | 请求头模板 |
| `body` | 请求体模板 |
| `success` | 成功条件，例如 `code equals 200` 或 `code equals 0` |
| `extract` | 从响应中提取变量，保存到运行时上下文 |
| `answerPaths` | 从响应中提取最终回答文本 |
| `skipWhenVariableExists` | 当某个变量已存在时跳过当前步骤 |
| `nullWhenMissingVariables` | 模板变量缺失时发送 JSON `null`，而不是空字符串 |

变量模板规则：

- `{{userMessage}}` 表示当前用户问题。
- `{{chat_id}}`、`{{sessionId}}` 等表示之前从第三方响应中提取并保存的变量。
- JSON 字段值如果完整等于单个模板，例如 `"sessionId": "{{sessionId}}"`，运行时会注入原始值；缺失时可配合 `nullWhenMissingVariables` 发送 JSON `null`。

## 方案 A：先创建会话，再发消息

适用形态：第三方提供独立的“打开会话 / 创建会话”接口，后续消息接口通过路径或请求体带会话 ID。

华鹏示例见 `wiki/第三方api.md`：

- `GET /application/{application_id}/chat/open` 返回 `data`，即 `chat_id`
- `POST /application/chat_message/{chat_id}` 发送消息

### 正确流程

1. 第一轮没有 `chat_id`。
2. 执行 `open_chat`，从响应 `$.data` 提取 `chat_id`。
3. 执行 `send_message`，路径里使用 `{{chat_id}}`。
4. 第二轮开始，发现 `chat_id` 已存在，跳过 `open_chat`。
5. 继续用同一个 `chat_id` 调用消息接口。

### 配置示例

```json
{
  "steps": [
    {
      "key": "open_chat",
      "method": "GET",
      "path": "/application/{{externalApplicationId}}/chat/open",
      "success": { "path": "$.code", "equals": 200 },
      "extract": [
        { "name": "chat_id", "path": "$.data" }
      ],
      "skipWhenVariableExists": "chat_id"
    },
    {
      "key": "send_message",
      "method": "POST",
      "path": "/application/chat_message/{{chat_id}}",
      "body": {
        "message": "{{userMessage}}",
        "stream": false
      },
      "success": { "path": "$.code", "equals": 200 },
      "answerPaths": ["$.data.content", "$.data.answer_list[0].content"]
    }
  ]
}
```

### 方案 A 检查点

- `open_chat` 必须有 `extract`，把第三方返回值保存成变量。
- `open_chat` 必须配置 `skipWhenVariableExists`，否则每轮都会重新创建会话。
- `send_message` 必须使用提取出来的变量，不能使用本地 `sessionId`。
- 日志中同一轮本地会话的 `chat_id` 应保持不变。

## 方案 B：单个 chat 接口自动创建或续接会话

适用形态：第三方只有一个便捷 chat 接口，首轮 `sessionId` 为空，响应返回 `sessionId`，后续请求把该值放回请求体。

FlowMesh 示例见 `wiki/第三方API 2.md`：

- `POST /apps/{slug}/chat`
- 首轮请求体 `sessionId: null`
- 响应 `data.sessionId`
- 后续请求体 `sessionId: 上一轮返回值`

### 正确流程

1. 第一轮没有 `sessionId`。
2. 请求体发送 `"sessionId": null`。
3. 从响应 `$.data.sessionId` 提取第三方 `sessionId`。
4. 第二轮开始，请求体发送 `"sessionId": "第三方返回的值"`。

### 配置示例

```json
{
  "steps": [
    {
      "key": "chat",
      "method": "POST",
      "path": "/apps/{{externalApplicationId}}/chat",
      "body": {
        "query": "{{userMessage}}",
        "sessionId": "{{sessionId}}"
      },
      "nullWhenMissingVariables": ["sessionId"],
      "success": { "path": "$.code", "equals": 0 },
      "extract": [
        { "name": "sessionId", "path": "$.data.sessionId" }
      ],
      "answerPaths": ["$.data.answer"]
    }
  ]
}
```

### 方案 B 检查点

- 请求体不能长期写死 `"sessionId": null`。
- 正确写法是 `"sessionId": "{{sessionId}}"` 加 `nullWhenMissingVariables: ["sessionId"]`。
- `extract` 必须从响应里保存第三方返回的 `sessionId`。
- `success.equals` 要按真实文档配置。FlowMesh 文档写的是 `code = 0` 成功；如果真实接口返回其他值，需要以实测为准并记录原因。
- 日志中同一轮本地会话的第三方 `sessionId` 应保持不变。

## 新厂商接入步骤

1. 先读厂商文档，找出“第三方上游会话 ID”的创建位置、响应路径和后续传入位置。
2. 确认首轮无上下文时，厂商要求字段缺失、空字符串还是 JSON `null`。
3. 在 `extract` 中保存厂商返回的上游会话 ID。
4. 在后续请求的 `path`、`body` 或 `headers` 中使用对应模板变量。
5. 如果存在独立创建会话步骤，配置 `skipWhenVariableExists`。
6. 如果首轮要求 JSON `null`，配置 `nullWhenMissingVariables`。
7. 用同一个本地设备/语音/WebSocket 会话连续问两轮，检查诊断日志里的第三方会话 ID 是否一致。

## 排障清单

### 日志看什么

重点看：

```text
[THIRD-PARTY-SESSION-DIAG] ... 变量名称：xxx 会话ID：yyy
```

同一个本地会话内：

- 第一轮允许出现新的第三方会话 ID。
- 第二轮及以后应该继续使用同一个第三方会话 ID。
- 如果每轮都变化，说明创建会话步骤未跳过，或请求体没有把旧 ID 传回第三方。

辅助看：

```text
[VOICE-SESSION-DIAG]
```

它用于确认本地运行时会话是否稳定。它稳定不代表第三方上下文一定稳定，只说明本系统能找到同一轮本地会话。

### 常见错误

| 现象 | 常见原因 | 修复方向 |
| --- | --- | --- |
| 方案 A 每轮 `chat_id` 都变 | 创建会话 step 每轮都执行 | 给创建会话 step 加 `skipWhenVariableExists` |
| 方案 A 第二轮路径里变量为空 | `extract.path` 配错 | 对照响应 JSON 修正提取路径 |
| 方案 B 每轮 `sessionId` 都变 | 请求体写死 `sessionId: null` | 改成 `"sessionId": "{{sessionId}}"` 并加 `nullWhenMissingVariables` |
| 日志里本地会话稳定但第三方 ID 变化 | 调用方没有传入可持久化的 `ChatConversation`，或集成配置覆盖了旧变量 | 检查 HTTP 语音、WebSocket realtime、后台调试入口是否都传入 conversation |
| 首轮把空字符串发给第三方导致报错 | 厂商要求 JSON `null` | 给该变量加入 `nullWhenMissingVariables` |
| 其他厂商上下文异常 | 硬编码了 `chat_id` / `sessionId` | 按厂商真实字段名配置变量，不要依赖固定名称 |

## 最低回归测试建议

每接入一种新形态，至少补一个两轮调用测试：

1. 模拟第三方第一轮返回上游会话 ID。
2. 第二轮断言请求第三方时带的是第一轮返回的 ID。
3. 断言本地 `ChatConversation.external_session` 保存了该变量。
4. 如果是 WebSocket 或设备语音入口，还要覆盖对应入口是否传入了同一个 conversation。

- `backend/apps/devices/tests/test_device_authorization_api.py`
- `backend/config/tests/test_realtime_websocket.py`

## 不要做的事

- 不要本地生成第三方 `chat_id`、`sessionId`、`threadId`。
- 不要把安卓或 WebSocket 的本地 `sessionId` 直接发给第三方。
- 不要把某个厂商字段名写死到通用逻辑里。
- 不要只看本地会话是否稳定；必须确认第三方上游会话 ID 也稳定。
- 不要为了兼容一个厂商破坏其他厂商的配置语义。
