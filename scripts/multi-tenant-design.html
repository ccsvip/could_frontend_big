① 目标与非目标

DO设计目标

  • · 账号申请通过 = 创建公司。申请人成为该公司唯一的「公司管理员」。
  • · 公司间数据 100% 隔离，没有任何业务数据跨公司共享。
  • · 公司管理员可在自己公司内 创建 / 停用 / 重置密码 员工账号。
  • · 员工登录后只能看到所属公司的数据，看不到任何其他公司的影子。
  • · superuser（平台运维）仍能登录 /admin/ 跨公司维护。
  • · 现有业务功能（设备 / 资源 / 知识库 / AI 聊天 / 控制指令 / 点位）全部纳入隔离边界。

DON'T非目标

  • · 不做跨公司"加好友 / 共享资源 / 邀请链接"等公开协作能力。
  • · 不做单 SSO（暂不接微信扫码 / 飞书 / 钉钉登录）。
  • · 不做计费 / 用量统计 / 套餐升级（这一版只做隔离骨架，计费可后置）。
  • · 不做"员工跨公司任职"。一个 user 只能属于一个 tenant。
  • · 不承诺"公司管理员可看后台 Django admin"。运维面板继续是 superuser 专属。

② 关键决策点（每条都要你拍板）

下面 7 个决策都会在本节给「候选 + 推荐 + 代价」。⭐ 是我推荐项，但你说了算。

D1 · 租户身份识别（用户登录时怎么知道是哪家公司）

最关键

这条决定登录页 UX、URL 形态、CORS / Cookie 边界。一旦定了很难改。

选项 A · 子域名隔离 acme.solin.com / bar.solin.com

每家公司独占子域；浏览器 Cookie / localStorage 天然隔离；视觉品牌强。
代价 通配符 SSL（*.solin.com）、DNS 自动化、反代规则、CORS 全要重写；本地开发要改 hosts。

选项 B · URL 路径前缀 /t/acme/devices

登录页 /t/acme/login 自带租户 hint；同一域名所有公司，零 DNS 改动。
代价 前端路由全部要套 /t/:slug/*，所有现有 14 个 view 路由要改；登录态切换要清楚 slug。

选项 C · 用户名/手机号全局唯一，登录时由后端反查 tenant ⭐ 推荐

登录页保持 用户名+密码 不变；后端 /auth/login/ 拿到 user 后从 user.tenant 装到 JWT claim；所有后续请求自动按 tenant 过滤。
优点 前端登录页 0 改动；URL/Cookie 不变。和当前申请审批流自然衔接（一个 user → 一个 tenant）。
代价 用户名"全局唯一"约束保留 — 公司A 已有 jack 时，公司B 不能再用 jack。员工创建时要前端友好地报"用户名已被占用"。

选项 D · 公司 code + 用户名 二段登录 [公司 code] [用户名] [密码]

登录字面上"进入自己的公司"。username 可以公司内唯一（不同公司可同名）。
代价 登录页要加一栏「公司 code」，员工记不住或拼错；和现有"申请审批 → 直接登录"流程脱节，需要给每家公司分配一个 code 并通知到所有员工；JWT 还得带 tenant_id。

需你确认 偏好选项 C（用户名全局唯一）还是 D（公司 code 二段登录）？字面"进入自己的公司"听起来像 D，但 C 落地最快、改动面最小，登录体验也是单输入框。

D2 · 数据隔离粒度

最关键

"100% 隔离"在工程上有三档实现，强度递增、运维成本递增。

方案实现隔离强度运维成本迁移工作量
行级隔离
⭐ 推荐		 所有业务表加 tenant_id 列；DRF ViewSet 基类自动 filter(tenant=request.tenant)；
Manager / QuerySet / Serializer.create 三处各一道防线。		 应用层强制 低 单库单 schema 大 14 张业务表 +
序列化器 + 视图集
schema-per-tenant django-tenants：每家公司一个 Postgres schema；中间件按 host/header 切 schema。 DB 层强制 高 每加一家公司要 migrate；备份 / 监控按 schema 拆 中
DB-per-tenant 每家公司独立 Postgres database。 物理强制 极高 库存 / 备份 / 连接池 N× 极大

推荐行级隔离的理由：当前已经是单 Postgres 单 schema 架构，行级方案不引入新的运维形态；防线放在 DRF 视图基类 + ORM Manager，配合代码审计就足够"100% 隔离"。等公司数到三位数、合规要求 schema 隔离时再切。

需你确认 行级隔离能不能接受？如果你的"100% 隔离"是合规/审计意义上的物理隔离，那就要做 schema-per-tenant，工作量翻倍但符合最高合规预期。

D3 · 公司管理员的产生方式

当前流程的最小修改 ⭐ 推荐

  1. 用户在 /login → 申请账号 提交：用户名 + 真名 + 企业名称 + 手机号 + 密码。
  2. superuser 在 Django admin /admin/accounts/accountapplication/ 审核。
  3. 审核通过触发：
    • 新建 Tenant(name=企业名称, code=auto-slug)
    • 新建 User(username=申请的用户名, password=申请时的哈希)
    • 把这个 User 设为 Tenant.admin_user，同时 User.tenant=新 Tenant
    • 给这个 User 自动绑 UserRole(role=tenant_admin)

这条路保留了"申请-审核-下发"的现有 UX，前端 / 邮件 / 飞书通知都不变。

替代方案：自助注册即开公司

省掉人工审核，一申请即开通。但容易被刷库 / 测试数据污染；和"账号申请审核"现有产品形态冲突。

D4 · 员工的产生方式

方案 A：公司管理员后台手工建号 ⭐ 推荐

前端给公司管理员加一个「员工管理」页 → 输入用户名/真名/手机号/初始密码 → 立即创建。员工拿到管理员告知的初始密码登录后强制改密。

最简、可控、不需要邮件/短信通道。

方案 B：管理员发邀请链接

管理员发一个一次性 token URL，员工访问后自己设密码。需要邮件/短信发送通道，先 skip 不做。

方案 C：员工自己注册（提交"想加入 XX 公司"申请）

公司管理员审批员工申请。流程长、容易和"申请开公司"混淆，PRD 不推荐。

需你确认 员工密码强制首次修改这个安全增强要做吗？做的话登录响应里要带 must_change_password 标志，前端拦截。

D5 · 公司内部角色 / 权限

方案 A：保持当前全局 Role 表，给每家公司预绑同一个 simple_user

公司管理员不能新建/编辑角色，所有员工权限完全一致。简单但僵硬。

方案 B：两层角色 — 平台预定义 + 公司可派生 ⭐ 推荐

Role 表加 tenant 字段（null = 平台预定义模板）。公司管理员复制一个模板成自己的 Role 树，可调菜单可调权限点。

列表 / 创建 / 修改 Role 的 API 自动按 tenant=request.tenant OR tenant IS NULL（只读） 过滤。

方案 C：完全公司独立 Role 表

Role 必须 tenant 非空，每家公司必须自己建角色。最干净但首单接入麻烦。

D6 · superuser 跨租户访问

方案 A：superuser 走 Django admin 跨租户；前端业务 API 仍按 tenant 过滤 ⭐ 推荐

/admin/ 自带跨租户视图（superuser 不属于任何 tenant，TenantManager 检测到 superuser 直接 .all()）。
前端业务 API 一视同仁 — superuser 用前端同样得选一家公司"impersonate"。

方案 B：superuser 在前端有"切换公司"下拉

JWT claim 里带 acting_tenant_id。功能强但增加前端复杂度。

D7 · 现有数据怎么办（迁移当下的 admin / solin / yrl / xthy / jack）

方案 A：不迁移，drop+重置

最干净。已有的 5 个用户和所有业务数据全部清空。开发期可以接受，生产已上线就不行。

方案 B：建一个「默认公司」收容现有数据 ⭐ 推荐

data migration 中：

  1. 建 Tenant(name='默认公司', code='default', is_legacy=True)
  2. 所有现有 User（除 superuser admin）tenant=默认公司
  3. 所有现有业务行（device/resource/knowledge/...）tenant=默认公司

用 Django data migration 一次性回填，可逆（reverse 把 tenant 字段空掉即可）。

③ 数据模型变更（基于 D1=C / D2=行级 / D5=两层角色 的推荐组合）

classDiagram class Tenant { +id: BigAuto +name: str +code: str (unique slug) +admin_user: FK User (one-to-one) +is_legacy: bool +is_active: bool +created_at } class User { +username (global unique) +password (hash) +is_active / is_staff / is_superuser +tenant: FK Tenant (nullable for superuser) } class AccountApplication { +username +applicant_name +enterprise_name +phone +password (hash) +reason +status +tenant: FK Tenant (set on approval) } class Role { +name +code +tenant: FK Tenant (null = 平台模板) +is_template: bool } class UserRole { +user: FK User +role: FK Role } class Device { +tenant: FK Tenant (NEW) +...原有字段 } class Resource { +tenant: FK Tenant (NEW) +... } class KnowledgeDocument { +tenant: FK Tenant (NEW) +... } class ChatConversation { +tenant: FK Tenant (NEW) +... } Tenant "1" --> "1" User : admin_user User "n" --> "1" Tenant : tenant AccountApplication "n" --> "0..1" Tenant Role "n" --> "0..1" Tenant UserRole "n" --> "1" User UserRole "n" --> "1" Role Device "n" --> "1" Tenant Resource "n" --> "1" Tenant KnowledgeDocument "n" --> "1" Tenant ChatConversation "n" --> "1" Tenant

关键约束：

  • User.tenant 为 null 当且仅当 is_superuser=True（DB CHECK 或 model clean）。
  • Tenant.admin_user 必须满足 admin_user.tenant_id == self.id（model clean）。
  • 所有业务表 on_delete=PROTECT 或 CASCADE—— 看 D8（公司删除策略，下面补一条）。

④ 鉴权流程时序

sequenceDiagram autonumber participant U as 员工浏览器 participant FE as React SPA participant API as DRF (uvicorn) participant MW as TenantMiddleware participant DB as Postgres U->>FE: 输入 username + password FE->>API: POST /auth/login/ API->>DB: User.objects.filter(username=...) DB-->>API: User { id, tenant_id, role_binding } API->>API: authenticate() → user API->>API: 校验 user.is_active && user.tenant.is_active API->>API: 签 JWT { sub, tenant_id, role_code } API-->>FE: { access, refresh, user, tenant } FE->>FE: store: { token, tenant, menus, permissions } Note over FE,API: 后续每次业务请求 FE->>API: GET /devices/ (Authorization: Bearer ...) API->>MW: decode JWT → request.tenant_id MW->>API: 注入 request.tenant API->>DB: Device.objects.filter(tenant=request.tenant) DB-->>API: 仅本公司的设备 API-->>FE: 200 OK

三道防线（任一层都能挡住跨租户读取）

  1. JWT：登录时把 tenant_id 写进 access token claim，篡改即签名失败。
  2. Manager：给所有 tenant-scoped model 加 TenantQuerySet，objects = TenantManager()；调用前必须 .for_tenant(t)，否则 RuntimeError（开发模式立刻爆炸，生产模式记 sentry）。
  3. ViewSet 基类：TenantScopedViewSetMixin.get_queryset() 强制 .for_tenant(self.request.tenant)；同时 perform_create 自动塞 tenant=self.request.tenant，前端就算偷偷传也覆盖掉。

⑥ 迁移策略（分 5 个 PR，渐进上线）

  1. PR-1：基础设施
    • 新增 apps/tenants/（Tenant 模型 + admin + serializer + 管理 API）
    • 新增 User.tenant 字段（null 允许）+ data migration 建「默认公司」+ 回填所有非 superuser
    • 新增 TenantMiddleware、TenantManager、TenantScopedViewSetMixin 工具类
    • JWT payload 增加 tenant_id claim
    • 测试覆盖：跨租户访问 → 404；同租户访问 → 200。
  2. PR-2：业务表批量加 tenant 字段
    • 14 张表逐张加 tenant FK（先可空，data migration 回填后改非空）
    • 所有 ViewSet 切到 TenantScopedViewSetMixin
    • 所有 Serializer.create 移除前端传入的 tenant 字段（防伪造）
    • 测试矩阵：每个 endpoint × 跨租户场景 = 必须 404
  3. PR-3：申请审批改造 + 自动建 Tenant
    • AccountApplication 审核通过 → 事务性创建 Tenant + 绑定 admin_user + 自动绑 tenant_admin Role
    • JWT 登录返回体增加 tenant: { id, name, code }
    • 前端 store 增加 tenant 信息
  4. PR-4：员工管理页（前端）
    • 新增 /employees 路由（PermissionGuard: tenant.employees.manage）
    • 列表 / 新建 / 停用 / 重置密码（CRUD）
    • 员工首次登录强制改密（如 D4 选要做）
  5. PR-5：Role 两层化（D5=B 时）
    • Role 加 tenant FK (nullable) + is_template
    • 预定义模板：tenant_admin / tenant_employee 两份
    • "复制模板成本公司角色" API + 前端

每个 PR 都独立可上线 / 可回滚。PR-1 / PR-2 上线后系统行为对最终用户来说零变化（只多了个看不见的"默认公司"），但已经具备了承接新公司的能力。

⑦ 风险点 & 对策

风险影响对策
R1 — 跨租户数据泄漏：开发者新加 ViewSet 忘了套 mixin P0：另一家公司能读到我家数据 ① CI 守护：lint 检查所有 ViewSet 必须继承 TenantScopedViewSetMixin（或显式标记 @tenant_unscoped）
② 测试矩阵：每个 endpoint 必有"跨租户 404"用例
R2 — Manager 漏网：直接用 Model.objects.all() 不带 for_tenant P0：管理脚本/admin action 跨租户改写 TenantManager 默认禁用裸 .all()，必须 .for_tenant(t) 或 .all_tenants()（明确旁路）
R3 — 媒体文件路径泄漏：知识库 PDF 上传到 /media/kb/uuid.pdf 后任意 tenant 拉到 P0：跨租户文件可被猜 URL 下载 媒体路径加 tenant code: /media/{tenant_code}/kb/...；下载视图二次校验 file.tenant == request.tenant
R4 — JWT 篡改 tenant_id 低：签名校验拦截 SimpleJWT 签名机制本身防篡改；登录时 tenant_id 直接来自 user.tenant 不可注入
R5 — 公司管理员被盗号 → 可改员工密码 → 整公司沦陷 中：单个公司 ① 强密码策略 + 改密日志
② 关键操作（重置员工密码）二次确认
③ 后置：增加 2FA
R6 — 默认公司收容大量历史数据，未来无法分拆 中：迁移期 预留 Tenant.is_legacy 标志，后续 superuser 可在 admin 手工把行迁移到正式公司
R7 — 容器内还有"company_name / password_hash" 孤儿列（来自另一个 worktree 的 in-progress 工作） 中：可能与本设计的 enterprise_name / password 冲突 本次 PR-1 之前必须先和那个 worktree 的负责人对账，要么补迁移要么删孤儿列。这是先决条件
