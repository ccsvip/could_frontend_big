📝 总体评估

📋 总体评估

项目概况：could_frontend 是面向数字人（数字员工）管理的全栈平台，后端 Django 5.2 + DRF + Celery，前端 React 18 + TypeScript + Vite。系统覆盖设备运行时配置、音色绑定、ASR/LLM/TTS 实时流式对话、第三方机器人上下文集成、租户隔离与权限管理。本次为全量（full）审计，覆盖 25 个维度，共确认 21 个发现（Critical 2、High 10、Medium 4、Low 3、Info 2），全部以 F1–F21 在详细发现展开。

主要风险：集中在安全与发布两条线。docker-compose 启动脚本内联创建 admin/admin123456 超管且无环境变量门控；Sentry DSN 硬编码且 send_default_pii=True 默认开；SECRET_KEY/ALLOWED_HOSTS/CORS 均为不安全默认值；CSRF 中间件被注释禁用。发布侧无 CI/CD，requirements 多数未固定，web 用 npm run dev 起生产，prod.py 仅 4 行未收紧。叠加意味着「按默认 compose 部署即获得可被秒破的超管账号」。

亮点：测试质量突出——49 个测试模块、250+ 用例、mock 计数为 0、跨租户隔离测试充分；ASGI 流式正确使用 httpx.AsyncClient；业务缓存命名空间化复用（DRY）；统一 WS 入口 + type 路由（KISS）；Manager.for_tenant 封装租户隔离（Law of Demeter）；DATABASE_URL 缺失 fail-fast（Fail-Fast 4.4）；前端 zustand store 组合子组件（Composition Over Inheritance）。

优先修复：立即修复 F1–F4（默认超管、Sentry PII、SECRET_KEY/ALLOWED_HOSTS/CORS、CSRF 中间件），4–6 小时消除最严重安全暴露；稳定发布前补齐 F5–F9、F11、F12（依赖固定、CI、realtime 拆分、_AGENT_MEMORY 治理、前端生产构建、prod 收紧、Sentry 仅 prod 初始化）；后续排期 F10、F13–F16。

总评：Overall 5.1 / B。平台工程基线扎实（测试、流式、租户隔离、统一 WS），但安全默认值与发布管线存在「屎山」级缺陷，不修复不建议进入稳定发布。

🔍 详细发现

F1. compose 内联创建 admin/admin123456 超管无门控

Critical Confirmed
📁 Security / Release / Configuration 📍 docker-compose.yaml:59 🎯 backend 启动命令
问题：backend 启动命令内联执行 python manage.py shell -c 创建 admin/admin123456 超管，无任何「首次部署 / 已存在超管 / 环境变量」门控。
为何重要：任何按默认 compose 部署的环境（含生产）都会获得 admin/admin123456 超管账号。该密码为常见弱口令，与 F15 弱密码校验叠加后用户改密仍可选弱密码。
失败场景：运维按默认 compose 在公网服务器 docker compose up -d，5173/80 暴露后 /admin/ 可用 admin/admin123456 直接登录，攻击者枚举/撞库即获超级权限。
docker-compose.yaml:59 backend command: python manage.py shell -c "from django.contrib.auth import get_user_model; User = get_user_model(); User.objects.filter(username='admin').exists() or User.objects.create_superuser('admin','admin@example.com','admin123456')" # 无首次部署 / 已存在超管 / 环境变量门控
最小修复
抽独立 create_default_superuser 管理命令，仅当 DJANGO_CREATE_DEFAULT_ADMIN=1 且 DJANGO_DEFAULT_ADMIN_PASSWORD 显式提供时才创建，并在创建后提示首次登录改密。
⏱ 工作量： 2–3h  ·  回归测试： 未设 DJANGO_CREATE_DEFAULT_ADMIN 时运行该命令不创建超管；开关=1 但缺密码应 fail-fast 退出非零。

F2. Sentry DSN 硬编码 + send_default_pii=True 默认开

Critical Confirmed
📁 Security / Privacy 📍 backend/config/settings/base.py:12-18 🎯 sentry_sdk.init
问题：sentry_sdk.init(dsn=硬编码DSN, send_default_pii=True, before_send=before_send)。DSN 整段写在源码，PII 默认开启。
为何重要：DSN 的 public+secret 都暴露在仓库，持有 secret 者可向项目投递伪造事件或滥用配额；send_default_pii=True 在 dev/prod 一律上报 IP、请求头、用户标识，违反数据最小化。
失败场景：DSN secret 随仓库克隆流出，攻击者用该 DSN 大量发送伪造事件淹没 Sentry 配额/告警，使真实告警被淹没；同时生产用户 IP/UA 被默认上报，触发 GDPR/PIPL 合规风险。
base.py:12-18 sentry_sdk.init( dsn="https://@sentry.io/", # 硬编码 send_default_pii=True, before_send=before_send, )
最小修复
DSN 移到 SENTRY_DSN 环境变量，未设则不 init；send_default_pii 由 SENTRY_SEND_DEFAULT_PII 显式开启，默认 False；before_send 内对 IP、User-Agent、Authorization 做 redaction。
⏱ 工作量： 1–2h  ·  回归测试： SENTRY_DSN 未设时 sentry_sdk.Hub.current.client is None；默认 SENTRY_SEND_DEFAULT_PII is False。

F3. SECRET_KEY/ALLOWED_HOSTS/CORS 不安全默认值

High Confirmed
📁 Security / Configuration 📍 base.py:31, :33, :248, :259 🎯 settings 默认值
问题：SECRET_KEY 默认 'django-insecure-change-me'；ALLOWED_HOSTS 默认 '*'；CORS_ALLOW_ALL_ORIGINS 默认 True；CORS_ALLOW_CREDENTIALS=True。
为何重要：任一未显式注入 env 的环境都用弱 key 签名 JWT/Session，可被伪造；ALLOWED_HOSTS=* 攥不住 Host 注入；CORS_ALLOW_ALL + ALLOW_CREDENTIALS 在 corsheaders 下回显具体 Origin，等于任意站点可携凭据跨域调 API。
失败场景：部署忘记注入 DJANGO_SECRET_KEY，系统用 django-insecure-change-me 静默启动，攻击者用该 key 伪造任意 Session/JWT 提权；恶意站点 JS 携 cookie 跨域调用本平台 API 读数据。
base.py:31 SECRET_KEY = os.getenv('DJANGO_SECRET_KEY','django-insecure-change-me') base.py:33 ALLOWED_HOSTS = ['*'] # 默认 base.py:248 CORS_ALLOW_ALL_ORIGINS = True # 默认 base.py:259 CORS_ALLOW_CREDENTIALS = True
最小修复
SECRET_KEY 缺失时 raise ImproperlyConfigured（与 DATABASE_URL 一致）；ALLOWED_HOSTS 默认空列表；CORS_ALLOW_ALL_ORIGINS 默认 False，需显式开启并配套白名单。
⏱ 工作量： 1h  ·  回归测试： 未设 SECRET_KEY 时 import settings 抛 ImproperlyConfigured；默认 CORS_ALLOW_ALL_ORIGINS is False。

F4. CSRF 中间件被注释禁用

High Confirmed
📁 Security 📍 base.py:97-109 🎯 MIDDLEWARE
问题：django.middleware.csrf.CsrfViewMiddleware 被注释禁用；REST_FRAMEWORK 的 DEFAULT_AUTHENTICATION_CLASSES 含 SessionAuthentication，admin 走 session。admin 原生 view 不走 DRF。
为何重要：禁掉 CSRF 中间件后 admin 表单提交不校验 token，跨站请求伪造可直接对 admin 增删改用户与数据。
失败场景：已登录 admin 的管理员访问恶意页面，该页面提交 POST /admin/auth/user/add/ 创建新超管，因无 CSRF 校验直接成功。
base.py:97-109 MIDDLEWARE = [ ... # django.middleware.csrf.CsrfViewMiddleware, # 被注释禁用 ... ] REST_FRAMEWORK['DEFAULT_AUTHENTICATION_CLASSES'] 含 SessionAuthentication # admin 走 session
最小修复
恢复 CsrfViewMiddleware（位于 AuthenticationMiddleware 之后、CommonMiddleware 之后，按 Django 推荐）。
⏱ 工作量： 1h  ·  回归测试： 不带 CSRF token 的 admin POST /admin/auth/user/add/ 应返回 403；带正确 token 返回 302。

F8. _AGENT_MEMORY 进程级全局无锁无 TTL 无界增长

High Confirmed
📁 Performance / Stability 📍 backend/config/realtime.py:43-44 🎯 _AGENT_MEMORY
问题：_AGENT_MEMORY: dict[str, list[dict[str,str]]] = {} 进程级全局；_get_agent_memory 直接 .get()，_push_agent_memory 用 setdefault().append() 再 del history[:-N]；无锁、无 TTL、无 LRU；每 device/agent 键永不清理。
为何重要：(1) 内存泄漏：每新会话键永不释放，worker RSS 单调增长至 OOM；(2) 并发：uvicorn 单 event loop 多协程可能同时 setdefault().append() 与 del history[:-N]，存在 RMW 竞态，极端丢消息或切片越界。
失败场景：上线数月后活跃设备/agent 键持续累积，worker RSS 缓慢增长至 OOM 被 kill，实时对话随机中断；高并发时段同一设备多协程写记忆出现丢消息。
realtime.py:43-44 _AGENT_MEMORY: dict[str, list[dict[str, str]]] = {} # 进程级全局 # _get_agent_memory: 直接 .get() # _push_agent_memory: setdefault().append() -> del history[:-N] # 无锁 / 无 TTL / 无 LRU / 每 device-agent 键永不清理
最小修复
用 cachetools.LRUCache(maxsize=N) 或自实现 LRU+TTL（30 分钟未访问淘汰）+ asyncio.Lock；或迁到 Redis 用 SETEX 自动过期 + 命名空间化（AGENTS.md 规定业务缓存走 business_cache）。
⏱ 工作量： 3–4h  ·  回归测试： 写 maxsize+1 个不同键后最老被淘汰；TTL 到期 get 返回空；并发 100 协程写同一键不丢消息。

F9. web 用 npm run dev 起生产

High Confirmed
📁 Release / Performance 📍 docker-compose.yaml:101-118 🎯 services.web

F14. ASR 上游 close fire-and-forget 无 timeout 无重试

Medium Confirmed
📁 Stability / Fallback 📍 backend/config/realtime.py:38-46 🎯 _close_asr_upstream_context_later
问题：用 asyncio.create_task 启动 _close_asr_upstream_context，内部 try/except Exception: logger.exception(...) 后返回，无 timeout、无重试、无 task 引用追踪；任务对象可能被 GC。
为何重要：ASR 上游 WS close 可能卡住（对端不回 close frame），create_task 无 wait_for 会悬挂协程；异常仅 log 无告警无重试，连接资源可能泄漏。
失败场景：ASR 上游网络异常不回 close frame，close 协程永久悬挂，随会话累积连接句柄耗尽，新会话无法建立上游连接。
realtime.py:38-46 async def _close_asr_upstream_context_later(context): task = asyncio.create_task(_close_asr_upstream_context(context)) # 无 timeout / 无重试 / 无 task 引用追踪 -> 可能被 GC # 内部: try/except Exception: logger.exception(...) 后返回
最小修复
改 asyncio.wait_for(context.__aexit__(None,None,None), timeout=5)，超时强制 await context.close()；保留 task 引用集合以便关闭时 cancel。
⏱ 工作量： 1h  ·  回归测试： mock __aexit__ 永不返回，调用后 5s 内 task 完成（超时路径）且无悬挂。
